# Milestone v1.1.0: Pipeline Consolidation & Storage Optimization

**Status:** SHIPPED 2026-03-21
**Phases:** 74-79
**Total Plans:** 21

## Overview

Eliminate 100GB+ of duplicate data by consolidating siloed tables into _u tables, generalize the 1D bar builder for extensible data source onboarding, and clean up pipeline debt.

## Milestone Summary

**Key Decisions:**
- Direct-to-_u writes, drop siloed tables (no archive period, DROP immediately)
- Generalized 1D bar builder with source registry (config, not code for new sources)
- Per-family rollout for _u migration (price_bars pilot first, then remaining)
- Row count verification sufficient (no shadow-write period)
- alignment_source CHECK constraints on all 6 _u tables; 5 valid values enforced at DB level
- TVC bars use venue_id=11 (TVC data source) not per-exchange venue_ids
- alignment_source for base table is 'multi_tf' not 'default'
- VACUUM FULL yielded 254 GB savings (-59%, 431 GB -> 177 GB)

**Issues Resolved:**
- Phase 78 gap closure: 5 runtime files still referenced dropped siloed tables after initial drops -- fixed in plans 78-04, 78-05, 78-06
- EMA ISO backfill: +3.3M rows needed in ema_multi_tf_u before retiring sync path
- AMA DELETE scope: alignment_source filter needed to prevent cross-source row deletion
- 8 tech debt items from milestone audit all cleaned

**Issues Deferred:**
- Multi-TF builder consolidation: builders are already source-agnostic, consolidation adds complexity without clear benefit
- New data source onboarding (e.g., Binance): BAR-03 enables this, but actually adding sources is a separate milestone
- EMA/AMA builder consolidation: keep separate -- different computation logic

**Technical Debt Incurred:**
- No alembic migration for 30-table DROP (manual SQL, acceptable for research system)
- Lazy imports in deferred MCP modules reference deleted client.py (zero active consumers)
- CPOOL has 735 stale VWAP bars from removed HL data (harmless)
