## V1 Success Criteria (from Project Plan)

These are the gates that must pass before v1.0.0 ships:

1. **Backtest:** Sharpe >= 1.0, Max DD <= 15%, turnover/fill realism documented
2. **Live paper:** 2+ weeks with tracking error < 1% to backtest and slippage < 50 bps
3. **Ops:** Kill switch functional, daily risk caps active, full logs, one-page post-trade report

## Requirements

### Strategy Selection & Validation (Research Track #1)

Use the v0.9.0 tooling (IC evaluation, PSR, purged K-fold) to run a proper strategy bake-off and select the 2 strategies for V1 paper trading.

- [ ] **STRAT-01**: Run IC evaluation on existing feature set (EMA derivatives, RSI, ATR, AMA indicators) across BTC/ETH on 1D timeframe; rank features by IC magnitude and stability (IC-IR)
- [ ] **STRAT-02**: Run walk-forward backtest on at least 3 signal types (EMA crossover, RSI mean-reversion, ATR breakout) with purged K-fold CV; report: Sharpe, Max DD, MAR, turnover per strategy
- [ ] **STRAT-03**: Select 2 strategies for V1 paper trading based on bake-off results; document rationale, parameter choices, and expected performance range
- [ ] **STRAT-04**: Final backtest of selected 2 strategies meets V1 criteria: Sharpe >= 1.0, Max DD <= 15%, with realistic fee/slippage assumptions documented

### Exchange Integration

Connect to one exchange for paper trading. Read-only market data first, then paper order submission.

- [ ] **EXCH-01**: Exchange adapter for one venue (e.g., Binance, Kraken, or Coinbase) with authenticated API access for BTC/ETH spot
- [ ] **EXCH-02**: Real-time (or near-real-time) price feed from exchange integrated into data pipeline — can compare live prices against DB bar data
- [ ] **EXCH-03**: Paper order submission API — adapter translates signal output into exchange-compatible order format without live execution

### Order & Fill Store

Persist every order, fill, and position change for full audit trail.

- [ ] **ORD-01**: `cmc_orders` table with order lifecycle tracking (created, submitted, filled, cancelled, rejected) via Alembic migration
- [ ] **ORD-02**: `cmc_fills` table linking fills to orders with exact execution price, fees, timestamp, and venue
- [ ] **ORD-03**: `cmc_positions` table tracking current position per asset with cost basis, unrealized PnL, and last update timestamp
- [ ] **ORD-04**: Order/fill/position writes are atomic — partial fill updates position correctly; rejected orders logged with reason

### Paper-Trade Executor

Engine that reads signals, places paper orders, and tracks execution.

- [ ] **EXEC-01**: Paper-trade engine reads from signal tables (`cmc_signals`) and generates orders based on selected strategy parameters
- [ ] **EXEC-02**: Simulated execution: paper fills at next-bar open (or exchange best bid/ask) with configurable slippage model
- [ ] **EXEC-03**: Position tracker: maintains current holdings, cost basis, and P&L per asset; updates on each fill
- [ ] **EXEC-04**: Execution loop runs on schedule (daily for 1D strategies) or on-demand; logs every decision (signal received, order generated, fill simulated, position updated)
- [ ] **EXEC-05**: Backtest parity mode: paper executor can replay historical signals to verify it produces identical results to the backtester

### Risk Controls

Implement the safety mechanisms required by the V1 deliverables.

- [ ] **RISK-01**: Kill switch: single command or API call that immediately flattens all paper positions and halts signal processing; logged with timestamp and reason
- [ ] **RISK-02**: Position caps: configurable maximum position size per asset (absolute and as % of portfolio); orders exceeding cap are rejected with log entry
- [ ] **RISK-03**: Daily loss stop: if portfolio drawdown from day-open exceeds configurable threshold (default: 3%), kill switch triggers automatically
- [ ] **RISK-04**: Discretionary override mechanism: manual position adjustments allowed but logged with user, reason, timestamp, and diff from system signal (per Vision Draft)
- [ ] **RISK-05**: Circuit breaker: if N consecutive signals result in losses, pause signal processing and alert (configurable N and loss threshold)

### Live/Backtest Drift Guard (Research Track #4)

Detect when paper trading results diverge from what the backtest predicted.

- [x] **DRIFT-01**: Parallel backtest runner: for each day of paper trading, run the same signals through the backtester on the same data window
- [x] **DRIFT-02**: Drift metrics computed daily: absolute P&L difference, tracking error (rolling), Sharpe divergence
- [x] **DRIFT-03**: Auto-pause trigger: if 5-day rolling tracking error exceeds 1.5% (configurable), kill switch activates and alerts sent
- [x] **DRIFT-04**: Drift report: weekly summary comparing paper vs backtest with attribution (fee model, slippage, timing, data differences)

### Loss Limits & Kill-Switch Policy (Research Track #2)

Define and validate the loss limit framework.

- [x] **LOSS-01**: Simulate day-level VaR across the 2 selected strategies using historical returns; determine appropriate daily loss cap per pool
- [x] **LOSS-02**: Simulate intraday stop scenarios: hard stop vs trailing stop vs time-stop; report recovery time and opportunity cost
- [x] **LOSS-03**: Define default pool-level caps for Conservative/Core/Opportunistic pools (per Vision Draft targets: C-Pool DD <= 10-12%, Core DD <= 20%)
- [x] **LOSS-04**: Override rules documented: when and how caps can be temporarily adjusted, approval requirements, logging

### Tail-Risk Policy (Research Track #3)

Evaluate and define tail-risk response.

- [x] **TAIL-01**: Backtest hard stops (fixed %) vs volatility-based position sizing across selected strategies; compare Sharpe, Max DD, and recovery time
- [x] **TAIL-02**: Define "flatten all risk" criteria: what market conditions trigger full exit (e.g., vol spike > N sigma, correlation breakdown, exchange halt)
- [x] **TAIL-03**: Tail-risk policy document: default sizing approach, flatten triggers, and escalation procedure

### Data Economics (Research Track #5)

Make the build-vs-buy decision for data infrastructure.

- [ ] **DATA-01**: Audit current data costs: CoinMarketCap API usage, DB storage, compute costs; document monthly run rate
- [ ] **DATA-02**: Compare vendor API + local cache (current model) vs data lake architecture; document TCO for each at current scale and projected 2x/5x scale
- [ ] **DATA-03**: Define trigger: at what cost or venue count does data lake investment become justified; document as decision record

### Perps Readiness (Research Track #6)

Paper-only checklist for perpetual futures capability.

- [ ] **PERP-01**: Funding rate capture: ingest funding rate history for BTC/ETH perps from one venue; persist in DB table
- [ ] **PERP-02**: Backtester extension: funding payments modeled in backtest P&L (long pays/receives funding per 8h period)
- [ ] **PERP-03**: Margin model: compute initial and maintenance margin requirements; track margin utilization in paper executor
- [ ] **PERP-04**: Liquidation buffer: define minimum margin buffer (e.g., 2x maintenance); alert at 1.5x; kill switch at 1.1x
- [ ] **PERP-05**: Venue downtime playbook: documented procedure for exchange halt/degradation during open positions

### Operational Dashboard (Live)

Extend the v0.9.0 Streamlit dashboard with live operational views.

- [ ] **DASH-L01**: Live PnL view: real-time (refreshing) portfolio value, daily P&L, cumulative P&L since inception
- [ ] **DASH-L02**: Exposure view: current positions per asset, notional value, % of portfolio, cost basis
- [ ] **DASH-L03**: Drawdown view: peak equity, current drawdown, max historical drawdown, time since peak
- [ ] **DASH-L04**: Drift view: paper vs backtest tracking error, displayed as rolling time series with alert threshold line
- [ ] **DASH-L05**: Risk status: kill switch state, daily loss consumed vs cap, position utilization vs caps, circuit breaker status

### V1 Validation

Run the 2-week paper trading validation and measure against success criteria.

- [ ] **VAL-01**: Paper trading runs for minimum 2 consecutive weeks with both selected strategies active
- [ ] **VAL-02**: Tracking error vs backtest measured: target < 1%
- [ ] **VAL-03**: Slippage measured and documented: target < 50 bps
- [ ] **VAL-04**: Kill switch tested: triggered manually and automatically (via daily loss stop) during validation period
- [ ] **VAL-05**: All operational logs reviewed: no unexplained gaps, no silent failures, full order/fill audit trail

### V1 Results Memo

The capstone deliverable: a formal report documenting V1 outcomes.

- [ ] **MEMO-01**: Methodology section: data sources, strategy descriptions, parameter selection process (IC/PSR/CV bake-off), fee/slippage assumptions
- [ ] **MEMO-02**: Results section: Sharpe ratio, Max DD, MAR, win rate, average trade, turnover — for both strategies, backtest and paper
- [ ] **MEMO-03**: Failure modes section: what went wrong, what would have gone wrong in worse conditions, stress test results
- [ ] **MEMO-04**: Research track answers: one-paragraph summary of findings for each of the 6 research tracks
- [ ] **MEMO-05**: Next steps: recommended V2 priorities based on V1 findings

## Out of Scope for v1.0.0

- Live trading with real capital — paper only
- Cross-venue arbitrage — single venue
- Options/derivatives beyond perps readiness checklist
- Multi-pool implementation — single portfolio for V1 (pool structure defined but not split)
- ML/AI strategy components — classical TA only (ML in Year 2-3 per vision)
- Cloud deployment — local/VM only
- Outside capital / fund structure — prop only
