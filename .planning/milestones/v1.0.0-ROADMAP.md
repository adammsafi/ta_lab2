# Milestone v1.0.0: V1 Closure — Paper Trading & Validation

**Status:** PLANNED (starts after v0.9.0 ships)
**Phases:** 42-55
**Prerequisite:** v0.9.0 Research & Experimentation (Phases 35-41) complete

## Overview

Close the V1 loop defined in the foundational Project Plan. Use the research tooling built in v0.9.0 (IC, PSR, purged K-fold, feature experimentation, dashboards) to select and validate 2 strategies, then build the paper-trading infrastructure to run them live and measure against the V1 success criteria: Sharpe >= 1.0, Max DD <= 15%, 2+ weeks paper trading with tracking error < 1%.

This milestone also closes all 6 research tracks from the Vision Draft and produces the V1 Results Memo.

## Phase Details

### Phase 42: Strategy Bake-Off
**Goal:** Use v0.9.0 research tooling to evaluate existing signals and select the 2 best strategies for V1 paper trading, with documented rationale and expected performance.
**Depends on:** v0.9.0 complete (IC evaluation, PSR, purged K-fold, feature experimentation all available)
**Requirements:** STRAT-01, STRAT-02, STRAT-03, STRAT-04
**Success Criteria:**
  1. IC scores computed for all features across BTC/ETH 1D; features ranked by IC-IR
  2. Walk-forward backtest results for >= 3 signal types with purged K-fold; no data leakage
  3. 2 strategies selected with documented rationale; parameters chosen via walk-forward, not in-sample optimization
  4. Final backtest meets Sharpe >= 1.0, Max DD <= 15% with realistic fees/slippage

---

### Phase 43: Exchange Integration
**Goal:** Connect to one exchange API for paper trading — authenticated access for BTC/ETH spot, price feed comparison, and paper order format translation.
**Depends on:** Phase 42 (need to know which strategies to paper-trade)
**Requirements:** EXCH-01, EXCH-02, EXCH-03
**Success Criteria:**
  1. Authenticated API connection to one exchange with BTC/ETH spot access
  2. Live price feed ingested and compared to DB bar data — discrepancies logged
  3. Paper order adapter translates signal output to exchange order format without submitting

---

### Phase 44: Order & Fill Store
**Goal:** Persist every order, fill, and position change in the database with full audit trail and atomic updates.
**Depends on:** Phase 43 (needs exchange order format for schema design)
**Requirements:** ORD-01, ORD-02, ORD-03, ORD-04
**Success Criteria:**
  1. `cmc_orders`, `cmc_fills`, `cmc_positions` tables created via Alembic
  2. Order lifecycle tracked: created -> submitted -> filled/cancelled/rejected
  3. Partial fills update position correctly with weighted cost basis
  4. All writes are atomic — no orphaned fills or inconsistent positions

---

### Phase 45: Paper-Trade Executor
**Goal:** Engine that reads signals, places paper orders, tracks positions and P&L, and can be verified against the backtester for parity.
**Depends on:** Phase 44 (needs order/fill store), Phase 46 (risk controls can develop in parallel)
**Requirements:** EXEC-01, EXEC-02, EXEC-03, EXEC-04, EXEC-05
**Success Criteria:**
  1. Executor reads signals and generates orders for selected strategies
  2. Paper fills simulated with configurable slippage model
  3. Position tracker maintains holdings, cost basis, unrealized P&L
  4. Execution loop runs daily with full logging
  5. Historical replay produces results matching backtester within tolerance

---

### Phase 46: Risk Controls
**Goal:** Implement kill switch, position caps, daily loss stops, circuit breaker, and discretionary override logging — the safety net required before running any paper trades.
**Depends on:** Phase 44 (needs position tracking), Phase 45 (needs executor to enforce controls)
**Requirements:** RISK-01, RISK-02, RISK-03, RISK-04, RISK-05
**Success Criteria:**
  1. Kill switch flattens all positions and halts processing in < 5 seconds
  2. Position caps enforced — oversized orders rejected with log
  3. Daily loss stop triggers kill switch when threshold exceeded
  4. Discretionary overrides logged with full audit trail
  5. Circuit breaker pauses on N consecutive losing signals

---

### Phase 47: Drift Guard
**Goal:** Continuously compare paper trading results against parallel backtests to detect execution drift before it compounds.
**Depends on:** Phase 45 (needs paper executor running)
**Requirements:** DRIFT-01, DRIFT-02, DRIFT-03, DRIFT-04
**Success Criteria:**
  1. Parallel backtest runs daily on same data window as paper executor
  2. Drift metrics (P&L diff, tracking error, Sharpe divergence) computed and persisted
  3. Auto-pause triggers when 5-day tracking error > 1.5%
  4. Weekly drift report with attribution (fees, slippage, timing, data)

---

### Phase 48: Loss Limits Policy
**Goal:** Research and define the loss limit framework for all three capital pools using simulation and historical analysis.
**Depends on:** Phase 42 (needs strategy selection and backtest results)
**Requirements:** LOSS-01, LOSS-02, LOSS-03, LOSS-04
**Plans:** 4 plans
Plans:
- [ ] 48-01-PLAN.md — Alembic migration (pool_name + override columns) + VaR simulator library
- [ ] 48-02-PLAN.md — Stop-loss simulator library (hard/trailing/time-stop sweep)
- [ ] 48-03-PLAN.md — VaR simulation CLI + stop simulation CLI + reports + charts
- [ ] 48-04-PLAN.md — Pool cap definition CLI + override governance validation + policy docs
**Success Criteria:**
  1. Day-level VaR simulated for both strategies; daily loss caps recommended
  2. Intraday stop scenarios compared (hard/trailing/time); trade-offs documented
  3. Pool-level caps defined for C-Pool (DD <= 10-12%), Core (DD <= 20%), Opportunistic
  4. Override rules documented with approval requirements

### Phase 53: V1 Validation
**Goal:** Run the full paper-trading system for 2+ weeks and measure against V1 success criteria.
**Depends on:** Phases 42-52 (everything must be running)
**Requirements:** VAL-01, VAL-02, VAL-03, VAL-04, VAL-05

**Goal:** Close the evaluation gap — run the v0.9.0 IC and experimentation infrastructure on real data, score all existing features and AMA variants, validate signal quality, and populate dashboards with empirical results.
**Depends on:** v0.9.0 complete (Phases 37-38 IC evaluation + feature experimentation framework already built)
**Requirements:** EVAL-01, EVAL-02, EVAL-03, EVAL-04, EVAL-05
**Success Criteria:**
  1. IC scores computed for all canonical features (rsi_14, EMA crossovers, vol ratios, return z-scores) across BTC/ETH 1D; features ranked by IC-IR

  5. Evaluation findings documented: top features by IC-IR, regime-conditional IC breakdown, promoted/deprecated feature decisions
