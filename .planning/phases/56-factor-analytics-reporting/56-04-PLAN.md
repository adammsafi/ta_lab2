    Implementation (~40-50 lines):

    1. argparse flags:
       - `--feature` (required): feature column name (e.g., 'rsi_14', 'ret_arith')
       - `--tf` (default '1D'): timeframe filter
       - `--asset` (optional int): asset_id filter. If omitted, average IC across all assets.
       - `--return-type` (default 'arith'): 'arith' or 'log'
       - `--output` (optional): output HTML path (default: reports/ic_decay/{feature}_{tf}.html)

    2. Main logic:
       a. Get engine via `get_engine()` with NullPool

          ```sql
          SELECT horizon, AVG(ic) as ic, AVG(ic_p_value) as ic_p_value, AVG(rank_ic) as rank_ic
          FROM public.cmc_ic_results
          WHERE feature = :feature AND tf = :tf AND return_type = :return_type
          [AND asset_id = :asset_id -- if --asset provided]
          GROUP BY horizon
          ORDER BY horizon
          ```
       c. Call `plot_ic_decay(ic_df, feature, return_type=return_type)` from `ta_lab2.analysis.ic`
       d. Save chart: `fig.write_html(output_path)` + `os.makedirs` for parent dir
       e. Print: feature name, n horizons, output path

    3. Follow existing CLI patterns (NullPool engine, logging setup, argparse).

    Module invocation: `python -m ta_lab2.scripts.analysis.run_ic_decay --feature rsi_14 --tf 1D`

    CRITICAL: This CLI must query cmc_ic_results (populated by the sweep), NOT recompute IC from scratch. It is a visualization tool, not a computation tool.
