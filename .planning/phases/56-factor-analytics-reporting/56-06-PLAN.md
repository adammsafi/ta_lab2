    Create `src/ta_lab2/scripts/features/refresh_cmc_cs_norms.py` CLI script that computes cross-sectional normalization for 3 pilot feature columns using PostgreSQL window functions.

    Pilot columns (from RESEARCH.md recommendations):
    - `ret_arith` -> `ret_arith_cs_zscore`, `ret_arith_cs_rank`
    - `rsi_14` -> `rsi_14_cs_zscore`, `rsi_14_cs_rank`
    - `vol_parkinson_20` -> `vol_parkinson_20_cs_zscore`, `vol_parkinson_20_cs_rank`

    SQL pattern for each source column (use a CTE for clarity):

    ```sql
    WITH cs_stats AS (
        SELECT
            id, ts, tf,
            {src_col},
            AVG({src_col}) OVER (PARTITION BY ts, tf) AS cs_mean,
            STDDEV({src_col}) OVER (PARTITION BY ts, tf) AS cs_std,
            PERCENT_RANK() OVER (PARTITION BY ts, tf ORDER BY {src_col}) AS cs_prank,
            COUNT(*) OVER (PARTITION BY ts, tf) AS n_assets
        FROM public.cmc_features
        WHERE {src_col} IS NOT NULL
    )
    UPDATE public.cmc_features f
    SET
        {src_col}_cs_zscore = CASE
            WHEN cs.n_assets >= 5 THEN (cs.{src_col} - cs.cs_mean) / NULLIF(cs.cs_std, 0)
            ELSE NULL
        END,
        {src_col}_cs_rank = CASE
            WHEN cs.n_assets >= 5 THEN cs.cs_prank
            ELSE NULL
        END
    FROM cs_stats cs
    WHERE f.id = cs.id AND f.ts = cs.ts AND f.tf = cs.tf;
    ```

    Implementation details:
    1. argparse with `--tf` (default '1D'), `--all-tfs` flag, `--dry-run` flag
    2. For each TF: run the UPDATE for all 3 source columns (6 target columns)
    3. Use `engine.begin()` transaction per TF
    4. Log: rows updated per column per TF, timing
    5. Use NullPool engine (one-shot connection pattern)
    6. CRITICAL: n_assets >= 5 guard prevents meaningless CS z-scores when few assets have data
    7. CRITICAL: NULLIF(cs_std, 0) prevents division by zero when all assets have same value

    Also expose a callable function `refresh_cs_norms(engine, tf: str) -> int` that can be called programmatically from the orchestrator (not just via CLI). The CLI `main()` calls this function internally.

    CRITICAL return type: `refresh_cs_norms()` MUST return `int` -- the sum of `cursor.rowcount` across all 3 UPDATE statements for the given TF. This is used by the orchestrator's `refresh_cs_norms_step()` as `rows_inserted` in RefreshResult. If it returns None, RefreshResult construction will fail. Example:
    ```python
    def refresh_cs_norms(engine, tf: str = "1D") -> int:
        total_rows = 0
        with engine.begin() as conn:
            for src_col in PILOT_COLUMNS:
                result = conn.execute(text(sql.format(src_col=src_col)), {"tf": tf})
                total_rows += result.rowcount
        return total_rows
    ```

    If `--all-tfs`: query `SELECT DISTINCT tf FROM public.cmc_features` and loop. Default: just '1D'.
    If `--dry-run`: print SQL but don't execute.

    Module invocation: `python -m ta_lab2.scripts.features.refresh_cmc_cs_norms --tf 1D`

    CRITICAL from RESEARCH.md Pitfall 8: CS-norm columns MUST be computed by this separate refresh script, NOT by the existing feature refresh pipeline. The existing pipeline's `_get_table_columns()` auto-discovery will see these columns but should not try to write them (they will remain NULL from the feature pipeline and get populated here).
