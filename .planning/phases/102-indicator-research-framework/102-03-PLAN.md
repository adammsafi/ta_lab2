**run_ic_sweep.py changes:**

1. Add import at top: `from ta_lab2.analysis.multiple_testing import log_trials_to_registry`

2. In `_ic_worker()` — find every call site where `save_ic_results(conn, all_ic_rows, ...)` is called (there are multiple: one for features-only sweep, one for AMA sweep, etc.). After each successful `save_ic_results()` call, add:
   ```python
   try:
       n_logged = log_trials_to_registry(conn, all_ic_rows, source_table="ic_results")
       _logger.debug("Logged %d trials to trial_registry", n_logged)

   ```
   The try/except ensures trial logging failures do not break the IC sweep itself. Use `_logger` (the existing module logger).

**IMPORTANT:** Do NOT change any existing function signatures. Do NOT change the save_ic_results call or its parameters. The trial logging is purely additive — it runs after save_ic_results succeeds, within the same connection/transaction.

**IMPORTANT:** log_trials_to_registry only inserts rows where horizon=1 AND return_type='arith' (it filters internally). This means the logging call gets ALL ic_rows but only a subset is actually inserted to trial_registry.

  <action>
Augment `classify_feature_tier()` in feature_selection.py to accept an optional `perm_p_value` parameter that gates tier assignment.

**Changes to classify_feature_tier():**

1. Add parameter: `perm_p_value: Optional[float] = None` (after existing parameters, before ic_ir_cutoff to maintain backward compatibility).

2. Add tier gating logic (per CONTEXT.md tiered gate decision):
   - If perm_p_value is None: existing logic unchanged (backward compatible — permutation test not yet run for this feature).
   - If perm_p_value is not None:
     - perm_p_value >= 0.15: force tier to 'archive' regardless of IC-IR (strong evidence of no signal)
     - perm_p_value in [0.05, 0.15): cap tier at 'watch' maximum (marginal evidence — cannot be 'active' or 'conditional')
     - perm_p_value < 0.05: no additional constraint (passes permutation gate, existing IC-IR logic determines tier)

3. Apply the perm_p_value gate AFTER the existing tier classification logic. This means:
   - Compute tier as before using existing IC-IR rules
   - Then apply the perm_p_value downgrade if applicable (can only downgrade, never upgrade)

4. Update the function's docstring to document the new parameter and gating behavior.

**Do NOT change the function signature in a breaking way.** The new parameter is Optional with default None, so all existing callers are unaffected.

**Do NOT auto-update dim_feature_selection from trial_registry.** Per CONTEXT.md decision, tier changes require explicit runs of the feature selection script.
  </action>
  <verify>
```python

# Existing behavior unchanged when perm_p_value=None
t1 = classify_feature_tier(ic_ir_mean=0.5, pass_rate=0.8, stationarity='stationary')

assert t4 == 'active'
```
