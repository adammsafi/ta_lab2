  <action>
Step 1: Investigate existing Qdrant setup before starting Docker Compose.

```bash
# Check if Qdrant is currently running
docker ps --filter "name=qdrant" --format "{{.ID}} {{.Names}} {{.Ports}} {{.Mounts}}"

# Check existing Docker volumes that might contain Qdrant data
docker volume ls --filter "name=qdrant"

# If existing Qdrant container uses a named volume, note its name
# The docker-compose.yml creates `qdrant-storage` — if data lives in a different
# volume, we may need to either:
# (a) rename the volume in docker-compose.yml to match, OR
# (b) create a Qdrant snapshot and restore
```

Step 2: Create docker/.env from template (copy .env.example, fill OPENAI_API_KEY from existing environment).

```bash
# If OPENAI_API_KEY is already in the environment, write it to docker/.env
echo "OPENAI_API_KEY=${OPENAI_API_KEY}" > docker/.env
```

Step 3: If existing Qdrant data uses a different volume name than `qdrant-storage`, update docker-compose.yml to reference that volume name as `external: true` to preserve data. If no existing volume (e.g., Qdrant was running with a bind mount or not in Docker), proceed with the new named volume.

Step 4: Start services.

```bash
docker compose -f docker/docker-compose.yml up --build -d
```

Step 5: Wait for health checks to pass, then validate endpoints.

```bash
# Wait for containers to be healthy (up to 60s)
timeout 60 bash -c 'until docker compose -f docker/docker-compose.yml ps | grep -q "healthy"; do sleep 2; done'

# Validate Qdrant is accessible
curl -sf http://localhost:6333/healthz

# Validate REST health endpoint
curl -sf http://localhost:8080/health

# Validate MCP endpoint responds (POST to MCP endpoint with initialize)
curl -sf -X POST http://localhost:8080/mcp/ \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","method":"initialize","params":{"protocolVersion":"2025-03-26","capabilities":{},"clientInfo":{"name":"test","version":"1.0"}},"id":1}'

# Validate REST stats endpoint
curl -sf http://localhost:8080/api/v1/memory/stats
```

Step 6: Verify memory count is non-zero (existing 3,763+ memories are accessible).

```bash
curl -sf http://localhost:8080/api/v1/memory/stats | python -c "
import sys, json
data = json.load(sys.stdin)
count = data.get('total_memories', 0)
print(f'Memory count: {count}')
assert count > 0, f'Expected memories but got {count} — Qdrant data migration needed'
"
```

If memory count is 0, this means the existing Qdrant data is in a different volume. Debug by checking `docker volume ls` and comparing with the standalone Qdrant container's volume. The fix is typically to update docker-compose.yml to use the existing volume name as external.
