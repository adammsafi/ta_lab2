**docker/Dockerfile:**
```dockerfile
FROM python:3.11-slim

WORKDIR /app

# Install system dependencies (gcc for C extensions)
RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc && \
    rm -rf /var/lib/apt/lists/*

# Copy project files
COPY pyproject.toml .
COPY src/ src/

# Install ta_lab2 with orchestrator extras + MCP deps
# NOTE: fastmcp pulls in the mcp SDK as dependency, do NOT install mcp separately
RUN pip install --no-cache-dir ".[orchestrator]" "fastmcp>=3.0.0" "uvicorn[standard]"

EXPOSE 8080

# Health check — hit the REST health endpoint
HEALTHCHECK --interval=15s --timeout=5s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8080/health')" || exit 1

CMD ["uvicorn", "ta_lab2.tools.ai_orchestrator.memory.server:app", \
     "--host", "0.0.0.0", "--port", "8080"]
```

**docker/docker-compose.yml:**
```yaml
services:
  qdrant:
    image: qdrant/qdrant:latest
    ports:
      - "6333:6333"
      - "6334:6334"
    volumes:
      - qdrant-storage:/qdrant/storage
    restart: unless-stopped
    healthcheck:
      test: ["CMD-SHELL", "python3 -c \"import urllib.request; urllib.request.urlopen('http://localhost:6333/healthz')\" || curl -sf http://localhost:6333/healthz || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 10s

  memory-server:
    build:
      context: ..
      dockerfile: docker/Dockerfile
    ports:
      - "8080:8080"
    environment:
      - QDRANT_HOST=qdrant
      - QDRANT_PORT=6333
      - QDRANT_SERVER_MODE=true
      - OPENAI_API_KEY=${OPENAI_API_KEY}
    env_file:
      - .env
    depends_on:
      qdrant:
        condition: service_healthy
    restart: unless-stopped

volumes:
  qdrant-storage:
```

Key design decisions:
- Docker named volume `qdrant-storage` (NOT bind mount) — avoids Windows POSIX incompatibility
- Build context is `..` (project root) so COPY can access pyproject.toml and src/
- Qdrant healthcheck uses curl OR python urllib (Qdrant image may not have curl)
- `env_file: .env` loads secrets from docker/.env (gitignored)
- QDRANT_HOST=qdrant uses Docker Compose service name resolution
- No `version:` key — Docker Compose v2 spec, version field is optional/deprecated

IMPORTANT: The existing Qdrant data may live in a different Docker volume from a standalone container. Before first `docker compose up`, the user should:
1. Check `docker volume ls` for existing qdrant volumes
2. If data exists in another volume, either rename it or create a snapshot/restore

Add a comment in docker-compose.yml noting this migration concern.

**Part A: .mcp.json**

Create `.mcp.json` in project root with Streamable HTTP transport (NOT SSE — SSE is deprecated per MCP spec 2025-03-26):

```json
{
  "mcpServers": {
    "ta-lab2-memory": {
      "type": "http",
      "url": "http://localhost:8080/mcp/"
    }
  }
}
```

This file is project-scoped and checked into version control. Claude Code reads it automatically on session start.

Key points:
- `"type": "http"` = Streamable HTTP transport (the recommended transport)
- URL points to the MCP mount path from server.py
- Trailing slash on `/mcp/` is important for correct sub-path routing
- Server name "ta-lab2-memory" matches the FastMCP name in mcp_server.py
- NOTE: The ROADMAP mentions ".claude/settings.json" for registration, but `.mcp.json` at project root is the correct and current standard for Claude Code MCP server registration. This is intentional.

**Part B: CLAUDE.md**

Create `CLAUDE.md` at project root. This is the standard file Claude Code reads for project-specific instructions. Include MCP memory usage guidance alongside key project context.

The file should contain:

1. **Project overview section** — Brief summary of ta_lab2 (trading system, 90+ DB tables, etc.)

2. **MCP Memory Server section** — The core guidance for when/how to query memories:
   - The ta-lab2-memory MCP server provides semantic search over 3,763+ project memories
   - Available tools: `memory_search`, `memory_context`, `memory_store`, `memory_stats`, `memory_health`, `list_categories`
   - When to use `memory_search`: Before starting unfamiliar tasks, when encountering unknown patterns, when you need historical context about why a decision was made
   - When to use `memory_context`: When building prompts that need project context (returns pre-formatted markdown)
   - When to use `memory_store`: After discovering important patterns, making architectural decisions, or resolving tricky bugs — store the learning for future sessions
   - When NOT to query: For routine tasks in well-understood areas, for simple code changes following existing patterns
   - Example queries: "How does EMA calculation work?", "What are the gotchas with vectorbt?", "Why did we choose Qdrant over ChromaDB?"

3. **Key conventions section** — Reference `.memory/MEMORY.md` for detailed project memory and conventions (critical gotchas, table patterns, etc.)

4. **Prerequisites section** — Note that the MCP server requires `docker compose -f docker/docker-compose.yml up -d` to be running

Keep the file concise (under 100 lines). It is guidance, not documentation. Also update `.memory/MEMORY.md` to add a cross-reference note in the "Auxiliary Memory Files" section pointing to CLAUDE.md for MCP usage guidance.
