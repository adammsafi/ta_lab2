Create a single Alembic migration with two operations: (A) create dim_data_sources table with seed data, (B) add CHECK constraints on alignment_source.

**A. dim_data_sources table:**

Use `op.execute()` with raw SQL (matching the dim_venues migration pattern). Schema:

```sql
CREATE TABLE public.dim_data_sources (
    source_key        TEXT PRIMARY KEY,
    source_name       TEXT NOT NULL UNIQUE,
    source_table      TEXT NOT NULL,
    venue_id          SMALLINT NOT NULL REFERENCES public.dim_venues(venue_id),
    default_venue     TEXT NOT NULL,
    ohlc_repair       BOOLEAN NOT NULL DEFAULT FALSE,
    has_market_cap    BOOLEAN NOT NULL DEFAULT FALSE,
    has_timehigh      BOOLEAN NOT NULL DEFAULT FALSE,
    id_loader_sql     TEXT,
    src_cte_template  TEXT NOT NULL,
    join_clause       TEXT,
    id_filter_sql     TEXT NOT NULL,
    ts_column         TEXT NOT NULL DEFAULT 'timestamp',
    conflict_columns  TEXT NOT NULL,
    src_name_label    TEXT NOT NULL,
    description       TEXT
);
```

Key column notes:
- `source_key`: Short identifier ('cmc', 'tvc', 'hl')
- `conflict_columns`: Comma-separated PK column list for ON CONFLICT clause (varies by source -- see RESEARCH.md Pitfall 1)
- `src_cte_template`: The full CTE SQL template with `{placeholder}` style substitution points
- `id_filter_sql`: WHERE clause fragment template for filtering by IDs (e.g., `s.id = ANY(%(ids)s)`)
- `ts_column`: Source timestamp column name ('timestamp' for CMC/TVC, 'ts' for HL)

Seed with 3 rows (CMC, TVC, HL) using the SQL templates extracted in Task 1. For the SQL template TEXT values, use dollar-quoting (`$$...$$`) in the INSERT to avoid escaping issues with single quotes inside SQL.

**B. alignment_source CHECK constraints:**

Before adding constraints, run a safety query to check for any unexpected alignment_source values in each _u table. Include this as a comment in the migration explaining the pre-check.

Add CHECK constraint to each of the 6 _u tables:

```sql
ALTER TABLE public.price_bars_multi_tf_u
ADD CONSTRAINT chk_price_bars_u_alignment_source
CHECK (alignment_source IN (
    'multi_tf',
    'multi_tf_cal_us',
    'multi_tf_cal_iso',
    'multi_tf_cal_anchor_us',
    'multi_tf_cal_anchor_iso'
));
```

Repeat for all 6 tables:
1. `price_bars_multi_tf_u` -> `chk_price_bars_u_alignment_source`
2. `ema_multi_tf_u` -> `chk_ema_u_alignment_source`
3. `ama_multi_tf_u` -> `chk_ama_u_alignment_source`
4. `returns_bars_multi_tf_u` -> `chk_returns_bars_u_alignment_source`
5. `returns_ema_multi_tf_u` -> `chk_returns_ema_u_alignment_source`
6. `returns_ama_multi_tf_u` -> `chk_returns_ama_u_alignment_source`

For `ema_multi_tf_u`: Research notes it may have 'unknown' default values. The migration should first check:
```sql
SELECT DISTINCT alignment_source FROM ema_multi_tf_u WHERE alignment_source NOT IN ('multi_tf', 'multi_tf_cal_us', 'multi_tf_cal_iso', 'multi_tf_cal_anchor_us', 'multi_tf_cal_anchor_iso')
```
If any 'unknown' values exist, UPDATE them to the correct value (likely 'multi_tf') before adding the CHECK. Include this remediation step in the upgrade() function.
