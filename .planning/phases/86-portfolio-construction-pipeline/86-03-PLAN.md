    Extend run_parity_check.py with a `--bakeoff-winners` flag that auto-discovers winning strategies from the bake-off results and runs parity checks for each.

    **New CLI flag**:
    - `--bakeoff-winners`: action="store_true", help="Auto-discover winning strategies from strategy_bakeoff_results and run parity for each"
    - When `--bakeoff-winners` is used, `--signal-id` becomes optional (not required). Update the parser: change `--signal-id` from `required=True` to `required=False, default=None`, and add a validation check in main(): if neither `--signal-id` nor `--bakeoff-winners` is provided, print error and exit 1.
    - `--slippage-mode` default changes to "fixed" when --bakeoff-winners is used (because historical replay has fill price differences)

    **Auto-discovery logic** (new function `_discover_bakeoff_winners(engine) -> list[dict]`):

    The `strategy_bakeoff_results` table has NO `is_winner` or `signal_id` columns. The actual schema is:
    - strategy_name TEXT, asset_id INTEGER, tf TEXT, params_json JSONB, cost_scenario TEXT, cv_method TEXT
    - sharpe_mean, sharpe_std, max_drawdown_mean, max_drawdown_worst, total_return_mean, cagr_mean
    - trade_count_total, turnover, psr, dsr, psr_n_obs, pbo_prob
    - fold_metrics_json JSONB, computed_at TIMESTAMPTZ, experiment_name VARCHAR(128)
    - UNIQUE: (strategy_name, asset_id, tf, params_json, cost_scenario, cv_method)

    Winner discovery approach -- rank by sharpe_mean, take top-1 per (strategy_name, asset_id):
    ```sql
    WITH ranked AS (
        SELECT
            strategy_name,
            asset_id,
            tf,
            sharpe_mean,
            ROW_NUMBER() OVER (
                PARTITION BY strategy_name, asset_id
                ORDER BY sharpe_mean DESC
            ) AS rn
        FROM strategy_bakeoff_results
        WHERE cv_method = 'CPCV'
          AND sharpe_mean IS NOT NULL
    )
    SELECT strategy_name, asset_id, tf, sharpe_mean
    FROM ranked
    WHERE rn = 1
    ORDER BY strategy_name, asset_id
    ```

    If no CPCV rows exist, retry with cv_method = 'PKF' as fallback.

    Then resolve signal_id from dim_signals using the strategy_name -> signal_type mapping.
    All AMA strategies (ama_momentum, ama_mean_reversion, ama_regime_conditional) map to signal_type='ema_crossover' -- this is a confirmed Phase 83 decision: AMA strategies reuse the EMA signal lifecycle.

    ```python
    # Strategy name to signal_type mapping (Phase 83 decision)
    _STRATEGY_SIGNAL_MAP: dict[str, str] = {
        "ama_momentum": "ema_crossover",
        "ama_mean_reversion": "ema_crossover",
        "ama_regime_conditional": "ema_crossover",
    }
    ```

    For each winner row:
    1. Look up signal_type = _STRATEGY_SIGNAL_MAP.get(strategy_name)
    2. If no mapping, log warning and skip (unknown strategy)
    3. Query: `SELECT signal_id FROM dim_signals WHERE signal_type = :signal_type LIMIT 1`
    4. Return list of dicts with keys: strategy_name, asset_id, signal_id, sharpe_mean

    **Multi-signal parity loop** (in main()):
    When `--bakeoff-winners` is set:
    1. Call `_discover_bakeoff_winners(engine)`
    2. If empty, print "No bakeoff winners found in strategy_bakeoff_results" and return 1
    3. Override slippage_mode to "fixed" if not explicitly set by user
    4. For each winner, call `checker.check(config_id=args.config_id, signal_id=winner["signal_id"], ...)`
    5. Print each report
    6. At end, print summary: N/M signals passed parity
    7. Exit 0 if ALL pass, exit 1 if ANY fail

    **CRITICAL**: The research noted that Phase 82's `_persist_results()` writes to `strategy_bakeoff_results`, NOT `backtest_trades`. The ParityChecker._load_backtest_trades() joins on `backtest_runs.signal_id`. If no backtest_trades exist for these signal_ids, parity will report 0 backtest trades (FAIL). This is expected behavior -- the parity check will correctly identify that the linkage gap exists. Log a clear message: "No backtest trades found for signal_id={X} (strategy={Y}). Phase 82 bake-off results are in strategy_bakeoff_results, not backtest_trades. A backtest run linking step may be needed."

    **CRITICAL**: When running with --bakeoff-winners, use `slippage_mode="fixed"` by default (as specified in CONTEXT.md: parity = same direction and timing, fill price secondary).
