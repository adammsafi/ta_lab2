  <action>
Create `src/ta_lab2/notifications/macro_alerts.py` implementing:

**MacroAlertManager class:**
- Constructor: `__init__(self, engine: Engine, cooldown_hours: int = 6)`
- The cooldown_hours is the throttle window -- macro regimes are sticky (change maybe 1-3 times per month), so 6 hours is generous. This naturally prevents noise.

**Core method: `check_and_alert(self, profile: str = "default") -> list[dict]`:**
1. Query latest 2 distinct dates from cmc_macro_regimes WHERE profile = :profile, ORDER BY date DESC LIMIT 2.
2. If fewer than 2 rows, log info and return empty list (no prior state to compare).
3. Compare dimension labels (monetary_policy, liquidity, risk_appetite, carry) and composite regime_key between the two dates.
4. For each changed dimension, call `_send_dimension_alert(dimension, old_label, new_label, current_row)`.
5. If composite regime_key changed, call `_send_composite_alert(old_key, new_key, old_state, new_state, current_row)`.
6. Return list of dicts describing all transitions found (for logging).

**Throttling: `_is_throttled(self, alert_type: str, dimension: str | None) -> bool`:**
- Query cmc_macro_alert_log: `SELECT 1 FROM cmc_macro_alert_log WHERE alert_type = :type AND (dimension = :dim OR (:dim IS NULL AND dimension IS NULL)) AND sent_at > NOW() - (INTERVAL '1 hour' * :hours) AND throttled = FALSE LIMIT 1`
- If row exists, return True (already alerted within cooldown).
- IMPORTANT: Do NOT use `INTERVAL ':hours hours'` -- PostgreSQL does not allow parameterized literals inside INTERVAL. Use `INTERVAL '1 hour' * :hours` instead, which multiplies the unit interval by the parameter.

**Alert logging: `_log_alert(self, alert_type, dimension, old_label, new_label, regime_key, macro_state, throttled)`:**
- INSERT INTO cmc_macro_alert_log with all fields. Use engine.begin() for auto-commit.

**Dimension alert: `_send_dimension_alert(self, dimension, old_label, new_label, current_row)`:**
- Check throttle. If throttled, log and record with throttled=True.
- Format message (HTML for Telegram):
  ```
  <b>Macro Regime Change: {dimension}</b>

  {old_label} -> {new_label}

  Current regime: {regime_key}
  Macro state: {macro_state}

  Key metrics:
  - VIX: {vix_value}
  - HY OAS: {hy_oas_value}
  - Fed Funds: {dff_value}
  - Net Liquidity Z: {net_liq_z}
  ```
- For key metrics: query the most recent row from fred.fred_macro_features to get vixcls, hy_oas_level, dff, net_liquidity_365d_zscore. IMPORTANT: The column is `hy_oas_level` (NOT `hy_oas`) -- confirmed in feature_computer.py. If query fails, show "N/A".
- Use severity="warning" for most changes. Use severity="critical" for risk_appetite->RiskOff or carry->Unwind transitions.
- Call `send_alert(title, message, severity)` from ta_lab2.notifications.telegram.
- Log the alert to cmc_macro_alert_log.

**Composite alert: `_send_composite_alert(self, old_key, new_key, old_state, new_state, current_row)`:**
- Similar to dimension alert but title is "Macro Regime Transition".
- Include all 4 current dimension labels.
- Severity="critical" if new_state in ("cautious", "adverse"), else "warning".

**Module-level convenience function:**
```python
def check_and_alert_transitions(engine: Engine, profile: str = "default", cooldown_hours: int = 6) -> list[dict]:
    """Convenience wrapper -- create manager and run check."""
    manager = MacroAlertManager(engine, cooldown_hours=cooldown_hours)
    return manager.check_and_alert(profile=profile)
```

**Graceful degradation:** Wrap all send_alert calls in try/except. If telegram.is_configured() returns False, log warning and still record to cmc_macro_alert_log (so dashboard can show transition history even without Telegram).

**Important:** The cmc_macro_alert_log table is created by Plan 01's Alembic migration. Since this plan runs in Wave 1 parallel with Plan 01, the code should handle the case where the table doesn't exist yet (catch OperationalError/ProgrammingError on insert and log a warning). At runtime after migration, this works normally.
