  Use MADR 4.0 format (Markdown Any Decision Records). This is the concise, formal decision record.

  Structure:
  ```
  # ADR-001: Data Infrastructure Architecture Choice

  **Date:** 2026-02-25
  **Status:** Accepted

  ## Context and Problem Statement

  ta_lab2 currently uses local PostgreSQL (~[measured size] across [N] tables) backed by manually downloaded CoinMarketCap JSON files. The project plans to expand from 2 crypto assets to 10+ assets including equities within 6 months, and to 50+ assets with hourly data within 18 months. We need to decide whether to stay on local PostgreSQL, migrate to a cloud data lake, or adopt a managed time-series platform.

  ## Decision Drivers

  * Current data API cost is $0 (file-based loads, no paid APIs)
  * Developer time ($200-800/month) is the dominant cost
  * PostgreSQL handles current scale (47M rows, [X]GB) comfortably
  * 2x scale (~235M rows) is viable with partitioning
  * 5x scale (~2.8B rows) exceeds PostgreSQL's comfortable operating range
  * CMC bulk download is undocumented and fragile
  * Equities expansion requires new data vendors regardless of infrastructure choice

  ## Considered Options

  * **Option A:** Stay on local PostgreSQL with file-based ingestion (current model)
  * **Option B:** Migrate to DIY cloud data lake (S3 + Parquet + DuckDB)
  * **Option C:** Migrate to managed TimescaleDB Cloud

  ## Decision Outcome

  Chosen option: **Option A (Stay on local PostgreSQL)** through 2x scale, because:
  1. Current TCO is lowest ($200-800/month, all developer time)
  2. No migration effort required
  3. PostgreSQL handles 2x scale (235M rows, 60GB) with basic partitioning
  4. Migration to Option C is low-effort (1-2 weeks) and can be triggered when quantitative thresholds are met

  **When to re-evaluate:** When any trigger from the Decision Trigger Matrix is met (see reports/data-economics/tco-model.md), or on calendar schedule every 6 months.

  ### Consequences

  * Good: Zero migration cost, zero infrastructure cost, full local control
  * Good: Developer time freed for feature development instead of migration
  * Bad: Manual CMC bulk download remains fragile (mitigate: document process, identify CoinGecko as fallback)
  * Bad: No cloud backup of data (mitigate: periodic pg_dump)
  * Neutral: Equities vendor (Alpaca/Polygon) decision is independent of infrastructure choice

  ## Dissenting View: Migrate to TimescaleDB Cloud Now

  Arguments for early migration to Option C:
  1. **Reduce operational risk early:** Local machine is single point of failure; TimescaleDB Cloud provides automated backups, HA
  2. **CoinGecko API now:** Replace fragile CMC file downloads with CoinGecko Analyst ($129/mo) immediately; API-driven ingestion is more reliable
  3. **Low migration cost:** 1-2 weeks, PostgreSQL-compatible -- the switching cost is low enough to "just do it"
  4. **Avoid partitioning work:** TimescaleDB hypertables handle time-series partitioning automatically; manual PostgreSQL partitioning at 2x scale is developer time that could be avoided

  Counter-argument: The additional $50-100/month cloud cost + $129/month CoinGecko API is real money for a project that currently costs $0 in infrastructure. The current model works. Migrating solves problems we don't have yet.

  ## Links

  * Full analysis: reports/data-economics/ (cost-audit.md, vendor-comparison.md, tco-model.md)
  * Decision Trigger Matrix: reports/data-economics/tco-model.md (see "Decision Trigger Matrix" section)
  ```
