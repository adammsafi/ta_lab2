<task type="auto">
  <name>Task 2: IC Staleness Monitor script</name>
  <files>src/ta_lab2/scripts/analysis/run_ic_staleness_check.py</files>
  <action>
Create `run_ic_staleness_check.py` as a standalone CLI script (invokable via `python -m ta_lab2.scripts.analysis.run_ic_staleness_check`).

**ICStalenessMonitor class** with these responsibilities:

1. **Load active-tier features** from `configs/feature_selection.yaml` (read the `active` list). Research recommends limiting to top 10 by IC-IR mean. Get the feature names.

2. **Load representative asset IDs** -- initially BTC (id=1) and ETH (id=1027). Accept --ids override. For each active feature + asset:

3. **Compute multi-window IC-IR** using `compute_rolling_ic()` from `ta_lab2.analysis.ic` at three windows:
   - short=30 bars (~1 month daily)
   - medium=63 bars (~1 quarter)
   - long=126 bars (~6 months)

   Load feature series via `load_feature_series(engine, asset_id, feature_name, tf='1D')`. Compute forward returns via `compute_forward_returns(close, horizon=1)`.

4. **Decay detection**: A feature is decaying when:
   - short IC-IR < 0.7 (IC_IR_STALENESS_THRESHOLD) AND
   - medium IC-IR < 0.7
   - Guard: if either is NaN (insufficient data), skip -- do NOT flag as decaying.

5. **BL weight halving**: For each decayed feature, INSERT into dim_ic_weight_overrides:
   ```sql
   INSERT INTO dim_ic_weight_overrides (feature, asset_id, multiplier, reason, created_at)
   VALUES (:feature, :asset_id, 0.5, :reason, now())
   ON CONFLICT ON CONSTRAINT uq_ic_weight_overrides DO NOTHING
   ```
   Reason string: f"IC decay: short={short_ir:.2f}, medium={medium_ir:.2f} < {threshold}"

6. **Throttled Telegram alert** for each newly decayed feature:
   - Check pipeline_alert_log for recent alert of type='ic_decay' with alert_key=feature_name within 24h cooldown
   - If not throttled: send WARNING alert via telegram.send_alert() with title "IC Decay Warning" and message showing the 3-window IC-IR values
   - Log to pipeline_alert_log regardless (throttled=True/False)
   - Follow MacroAlertManager._is_throttled() and _log_alert() patterns exactly

7. **CLI interface** using argparse:
   - `--db-url` (optional, falls back to resolve_db_url())
   - `--ids` (default "1,1027" for BTC+ETH)
   - `--dry-run` (compute but don't write overrides or send alerts)
   - `--verbose` (logging.DEBUG)
   - `--threshold` (default 0.7)

   Return code: 0 = no decay, 1 = error, 2 = decay detected (for pipeline stage runner)

**Constants:**
- IC_IR_STALENESS_THRESHOLD = 0.7
- IC_IR_ACTIVE_CUTOFF = 1.0
- WINDOWS = {"short": 30, "medium": 63, "long": 126}
- COOLDOWN_HOURS_IC_DECAY = 24
- TIMEOUT for this stage: 300s (5 min)

**Important patterns to follow:**
- Use `from ta_lab2.scripts.refresh_utils import resolve_db_url` for DB URL resolution
- Use `sqlalchemy.create_engine(db_url)` for engine creation
- Wrap all DB operations in try/except with proper logging
- `encoding='utf-8'` on any file reads (Windows gotcha)
- Guard NaN with `math.isnan()` before any comparison
