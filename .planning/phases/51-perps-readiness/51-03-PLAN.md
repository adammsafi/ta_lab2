    Write a comprehensive operational playbook covering all downtime scenarios. Follow Phase 49 pattern (human-readable procedure document).

    Document structure:

    **1. Overview**
    - Purpose: procedure for handling venue outages during perpetual futures paper trading
    - Scope: 6 venues (Binance, Hyperliquid, Bybit, dYdX, Aevo, Aster)
    - Reference: venue_health_config.yaml for machine-readable thresholds

    **2. Health Status Definitions**
    - HEALTHY: All checks passing. Normal operations.
    - DEGRADED: Elevated latency, stale orderbook, or widened spreads. New orders paused if persistent. Monitor at 60s intervals.
    - DOWN: Service unavailable. Halt all orders on this venue immediately.

    **3. Downtime Types & Detection**
    Table covering each downtime type with detection method and typical duration:
    - API Failure (HTTP 5xx, connection timeout): health endpoint returns error or times out
    - Degraded Performance (slow responses >Nms): health endpoint response time exceeds max_latency_ms
    - Stale Orderbook: price data not updating for >stale_orderbook_seconds
    - Scheduled Maintenance: pre-announced via status page or API announcement
    - Regulatory Halt: exchange suspends trading (detected via trade suspension API or announcement)
    - Unusual Spread Widening: bid-ask spread exceeds spread_alert_pct threshold

    **4. Immediate Response Procedure**
    Step-by-step for each health state transition:
    - Healthy -> Degraded: Log, alert Telegram, monitor at 60s, do NOT halt
    - Degraded -> Down (or Degraded > 30 min): Halt new orders on venue, alert Telegram
    - Down with Open Positions: Execute hedge procedure (Section 5)
    - Down > 60 min: Trigger kill switch via CLI (`python -m ta_lab2.scripts.risk.kill_switch_cli activate --reason "venue_down:{venue}"`)

    **5. Hedge-on-Alternate-Venue Procedure**
    Step-by-step manual procedure:
    1. Identify open positions on down venue (query cmc_perp_positions)
    2. Select alternate venue from priority list (venue_health_config.yaml -> hedge_procedure.venue_priority)
    3. Verify alternate venue is HEALTHY
    4. Place opposing position on alternate venue (same size, opposite side)
    5. Log hedge order with reason="venue_downtime_hedge"
    6. Monitor both positions until primary venue recovers
    7. Unwind hedge when primary venue returns to HEALTHY
    Note: V1 is manual. Automated routing deferred.

    **6. Recovery Procedure**
    - Verify venue returns to HEALTHY status (3 consecutive health checks pass)
    - Review any missed funding payments during downtime
    - Unwind hedges placed on alternate venues
    - Resume normal order flow
    - Document incident in cmc_risk_events

    **7. Monitoring Checklist**
    Daily operational checks:
    - [ ] All 6 venue health endpoints responding
    - [ ] Funding rate ingest completed for all venues
    - [ ] No stale orderbook alerts in last 24h
    - [ ] Spread within normal range for all venues
    - [ ] cmc_perp_positions margin utilization within safe range

    **8. Reference**
    - venue_health_config.yaml: machine-readable thresholds
    - Kill switch CLI: `python -m ta_lab2.scripts.risk.kill_switch_cli`
    - Risk engine: src/ta_lab2/risk/risk_engine.py
    - Funding refresh: `python -m ta_lab2.scripts.perps.refresh_funding_rates`
