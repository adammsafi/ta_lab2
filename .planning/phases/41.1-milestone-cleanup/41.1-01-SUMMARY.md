# Phase 41.1 Plan 01: AMA Table Allowlist + Filter Support in ExperimentRunner Summary

Wire AMA tables into ExperimentRunner allowlist and extend _load_inputs with parameterized filter support, then register kama_er_signal and ama_ret_momentum as experimental features.

## What Was Built

### Task 1 â€” AMA tables in _ALLOWED_TABLES + filter-aware _load_inputs (runner.py)

Added 4 AMA tables to the `_ALLOWED_TABLES` frozenset:
- `cmc_ama_multi_tf`
- `cmc_ama_multi_tf_u`
- `cmc_returns_ama_multi_tf`
- `cmc_returns_ama_multi_tf_u`

Extended `_load_inputs` to read an optional `filters` dict from each input entry. Each key-value pair becomes a parameterized `AND "key" = :filter_key` clause appended to the SQL string after the existing `WHERE id / tf / ts` block. Filter keys are validated with the existing `_is_safe_identifier` helper before interpolation.

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| `filter_` prefix on SQLAlchemy param names | Prevents collision with existing `id`, `tf`, `start`, `end` params |
| Double-quote filter key in SQL | Handles reserved words (e.g., a future `timestamp` filter key) safely |
| Reuse `_is_safe_identifier` for filter keys | Consistent security boundary with existing column name validation |
| Do NOT add AMA tables to `_TABLES_WITHOUT_TF` | AMA tables have a `tf` column; they follow the standard WHERE clause |
