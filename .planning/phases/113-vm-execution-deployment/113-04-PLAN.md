Create the results pull script following the SAME direction as sync_hl_from_vm.py (VM->local). This is the familiar pattern.

**Tables to sync (8 execution state tables):**

| Table | Watermark Column | PK | Notes |
|-------|------------------|----|-------|
| orders | created_at | order_id | All order lifecycle data |
| fills | filled_at | fill_id | Fill prices, quantities |
| positions | updated_at | position_id | Current position state |
| paper_orders | created_at | paper_order_id | Paper order log |
| executor_run_log | started_at | run_id | Run summaries |
| drift_metrics | computed_at | (id, metric_name, ts) | Drift detection results |
| risk_events | created_at | event_id | Risk gate activations |
| order_events | created_at | event_id | Order state transitions |

**Core sync flow (mirrors sync_hl_from_vm.py exactly):**
1. For each table, query local DB for max watermark column value.
2. SSH to VM: `COPY (SELECT * FROM {table} WHERE {wm_col} > '{watermark}') TO STDOUT WITH CSV HEADER`.
3. Write to local temp file.
4. Load temp file into local staging table.
5. Upsert from staging to target: `INSERT INTO {table} SELECT * FROM _staging ON CONFLICT ({pk}) DO UPDATE SET ...` (update all non-PK columns).

**CLI args:**
- `--full` : full resync
- `--dry-run` : report VM row counts without syncing
- `--table <name>` : sync one table only
- No args: incremental sync of all 8 tables

**Watermark strategy:** Use a local `sync_results_watermarks` table (same pattern as sync_hl_from_vm.py uses hyperliquid.sync_log). Create if not exists.

**Idempotency:** ON CONFLICT DO UPDATE ensures re-runs are safe. Use the same _ssh_cmd(), _vm_psql(), _vm_copy_to_stdout() helper functions from sync_hl_from_vm.py (copy the pattern, not import -- different module).

**Scheduling note:** Designed to run on local cron every 4-6 hours per CONTEXT.md. Add a comment at the top of the file showing the suggested crontab entry:
```
# Suggested crontab: 0 */4 * * * cd /path/to/ta_lab2 && python -m ta_lab2.scripts.etl.sync_results_from_vm
```

**Error handling:** Same as sync_signals_to_vm -- one table failure logs error, continues to next, exit code 1 if any failed.
