**B. Create `src/ta_lab2/scripts/backtests/run_mass_backtest.py`:**

This is a new CLI script that orchestrates mass backtesting with resume-safe state tracking.

Module docstring showing usage:
```python
"""
Resume-safe mass backtest orchestrator.

Wraps BakeoffOrchestrator with mass_backtest_state table for tracking
completion of each (strategy, asset, params_hash, tf, cost_bps) combination.
Uses expanded grids from configs/mass_backtest_grids.yaml.

Usage:
    # Dry run: show planned combinations
    python -m ta_lab2.scripts.backtests.run_mass_backtest --dry-run

    # Full run with 4 workers
    python -m ta_lab2.scripts.backtests.run_mass_backtest --workers 4

    # Resume interrupted run (skip completed rows)
    python -m ta_lab2.scripts.backtests.run_mass_backtest --resume --workers 4

    # Run specific strategies only
    python -m ta_lab2.scripts.backtests.run_mass_backtest --strategies ema_trend rsi_mean_revert

    # CTF threshold strategies only (uses CTF data loader)
    python -m ta_lab2.scripts.backtests.run_mass_backtest --strategies ctf_threshold

    # All assets discovered from features table
    python -m ta_lab2.scripts.backtests.run_mass_backtest --all-assets --workers 4

    # Specific exchange cost matrix
    python -m ta_lab2.scripts.backtests.run_mass_backtest --exchange all --workers 4
"""
```

Key components:

1. **Grid loading from YAML:**
   ```python
   def _load_mass_grids(yaml_path="configs/mass_backtest_grids.yaml"):
       """Load expanded param grids from YAML. Returns dict: strategy_name -> list of param dicts."""
   ```
   Parse the YAML file. Each key is a strategy name, value has a `params` list.

2. **params_hash computation:**
   ```python
   def compute_params_hash(params: dict) -> str:
       """Compute stable 16-char hex hash of sorted params JSON."""
       import hashlib
       return hashlib.md5(json.dumps(params, sort_keys=True).encode()).hexdigest()[:16]
   ```

3. **State table interaction:**
   - `_load_completed_keys(engine) -> set`: Load all (strategy_name, asset_id, params_hash, tf, cost_bps) tuples with status='done'.
   - `_mark_running(engine, key_tuple)`: Upsert with status='running'.
   - `_mark_done(engine, key_tuple)`: UPDATE status='done', completed_at=now().
   - `_mark_error(engine, key_tuple, error_msg)`: UPDATE status='error', error_msg=msg.

4. **Orchestration loop:**
   The main loop iterates over strategies from the YAML grid. For each strategy:
   - Get signal_fn from REGISTRY via `get_strategy(name)`.
   - Determine if this strategy needs AMA loader (`ama_momentum`, `ama_mean_reversion`, `ama_regime_conditional`) or CTF loader (`ctf_threshold`) or standard loader.
   - Build BakeoffOrchestrator with the appropriate config.
   - For CTF strategies: extract unique `feature_col` values from the param grid to pass to `load_strategy_data_with_ctf()`.
   - Call orchestrator.run() with the strategy's param grid, passing appropriate loader via `ama_features` (for AMA) or by pre-loading CTF data.

   IMPORTANT: The BakeoffOrchestrator already handles deduplication via `_batch_existing_keys()` checking `strategy_bakeoff_results`. The mass_backtest_state table provides an ADDITIONAL resume layer for the mass orchestrator itself (tracking which strategy/asset combos have been fully processed, not just individual result rows).

   The approach: iterate over (strategy, asset_ids) pairs. For each pair, check mass_backtest_state. If all combos for that pair are done, skip. Otherwise, call orchestrator.run() for that strategy+asset. After completion, mark all combos as done.

   Actually, the most practical approach: call BakeoffOrchestrator.run() for each strategy across all assets (it already handles per-combo dedup). Then batch-insert state rows for completed combos. The resume flag just loads completed keys to skip entire (strategy, asset) pairs.

5. **CTF strategy handling:**
   For `ctf_threshold` strategy, the signal function needs CTF columns in the DataFrame. Since BakeoffOrchestrator calls `load_strategy_data()` internally, we need to handle this. Two options:
   - Option A: Extend orchestrator to accept a custom data loader function.
   - Option B: Run CTF strategies separately with manual data loading.

   Use Option B (simpler, avoids modifying BakeoffOrchestrator): For CTF strategies, iterate over assets manually. Load data via `load_strategy_data_with_ctf()`, then call `run_purged_kfold_backtest()` and `run_cpcv_backtest()` directly (same functions the orchestrator uses internally). Persist results via the orchestrator's `_persist_results()` method.

6. **CLI arguments:**
   - `--dry-run`: Print planned combos and exit.
   - `--resume`: Skip completed combinations from mass_backtest_state.
   - `--strategies`: List of strategy names to run (default: all from YAML).
   - `--assets`: Specific asset IDs. Default: all from features table.
   - `--all-assets`: Discover all asset IDs from features table.
   - `--tf`: Timeframe (default: 1D).
   - `--exchange`: Cost matrix selection (kraken, hyperliquid, all). Default: all.
   - `--workers`: Parallel workers (default: 1).
   - `--cpcv-top-n`: CPCV top-N filter (default: 3, meaning only top 3 param sets by PKF Sharpe get CPCV).
   - `--grids-yaml`: Path to grid YAML (default: configs/mass_backtest_grids.yaml).
   - `--experiment-name`: Lineage tag (default: "phase99-mass-v1").
   - `--verbose`: DEBUG logging.

7. **DSR handling:**
   At 460K runs, DSR is inflated (Pitfall 1 from research). Set `dsr = NULL` for mass runs. Log this limitation at startup:
   ```python
   logger.warning(
       "DSR computation is limited at scale (>1000 runs per group). "
       "DSR values may be inflated. See Phase 99 research notes."
   )
   ```
   BakeoffOrchestrator already computes DSR per asset group, which is fine. No changes needed — the per-asset-group DSR will be meaningful.

8. **Progress tracking:**
   After each strategy completes, log:
   ```
   [strategy_name] Completed: X assets, Y results persisted
   ```
   After all strategies, log total and update mass_backtest_state.

Use NullPool for the engine. Follow the same argparse pattern as run_bakeoff.py.

Create `src/ta_lab2/scripts/backtests/backfill_mc_bands.py`:

A post-run script that populates `backtest_metrics.mc_sharpe_lo/hi/median` for all rows where these columns are NULL. Uses `monte_carlo_trades()` from `analysis/monte_carlo.py`.

Module docstring:
```python
"""
Backfill Monte Carlo Sharpe confidence intervals on backtest_metrics.

Populates mc_sharpe_lo, mc_sharpe_hi, mc_sharpe_median for every backtest_metrics
row where these columns are NULL. Loads trade-level PnL from backtest_trades,
runs 1000 bootstrap resamples, and writes back the results.

Usage:
    python -m ta_lab2.scripts.backtests.backfill_mc_bands
    python -m ta_lab2.scripts.backtests.backfill_mc_bands --dry-run
    python -m ta_lab2.scripts.backtests.backfill_mc_bands --batch-size 100
"""
```

Logic:
1. Query all `backtest_metrics.run_id` where `mc_sharpe_lo IS NULL`.
2. For each run_id (in batches of --batch-size, default 500):
   a. Load `backtest_trades` for that run_id: `SELECT pnl_pct FROM public.backtest_trades WHERE run_id = :run_id`.
   b. Call `monte_carlo_trades(trades_df, n_samples=1000, seed=42)`.
   c. If result has non-None values, UPDATE backtest_metrics with mc_sharpe_lo, mc_sharpe_hi, mc_sharpe_median.
   d. If < 10 trades (returns None), set mc_sharpe_lo = mc_sharpe_hi = mc_sharpe_median = NULL but add a sentinel flag (set mc_n_samples = 0) so we don't retry.
3. Log progress: every 100 run_ids, log count processed and remaining.
4. Use NullPool engine. `encoding='utf-8'` for any string operations.

CLI args:
- `--dry-run`: Count NULL rows without processing.
- `--batch-size`: Number of run_ids to process per commit (default 500).
- `--verbose`: DEBUG logging.

IMPORTANT: `monte_carlo_trades()` expects a DataFrame with a `pnl_pct` column in PERCENTAGE points (e.g. 2.5 for +2.5%). The `backtest_trades.pnl_pct` column stores values in this format already (see 071_backtest_trades.sql comment). No conversion needed.

After backfill, log the final count of NULL vs non-NULL mc_sharpe_lo rows.
