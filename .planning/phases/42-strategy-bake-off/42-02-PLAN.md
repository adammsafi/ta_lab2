<objective>
Build the walk-forward backtest orchestrator and run all signal strategies through purged K-fold CV with the full Kraken cost matrix, producing out-of-sample metrics for composite scoring.

Purpose: This is the core evaluation engine. Without walk-forward OOS metrics, strategy selection would be based on in-sample performance (overfitting). The purged K-fold ensures no data leakage.
Output: `bakeoff_orchestrator.py` library + Alembic migration for results table + `run_bakeoff.py` CLI + OOS metrics persisted to DB.

NOTE on expanding-window re-optimization: 42-CONTEXT.md mentions "Run BOTH fixed-parameter and expanding-window re-optimization per fold." This plan implements fixed-parameter evaluation only. Expanding-window re-optimization is DELIBERATELY DEFERRED to a follow-up plan. Rationale: Fixed-parameter walk-forward is the standard baseline and sufficient for V1 strategy selection. Adding expanding-window introduces parameter re-optimization logic that doubles complexity and should be validated separately. Executors should NOT add expanding-window ad-hoc -- it should be a planned, tested addition.
</objective>

**Part A: Alembic migration for strategy_bakeoff_results table**

Create a new Alembic migration (run `alembic revision -m "add strategy_bakeoff_results"` to get the file). The migration creates one table:

```sql
CREATE TABLE IF NOT EXISTS public.strategy_bakeoff_results (
    id SERIAL PRIMARY KEY,
    strategy_name TEXT NOT NULL,         -- e.g. 'ema_trend', 'rsi_mean_revert', 'breakout_atr'
    asset_id INTEGER NOT NULL,
    tf TEXT NOT NULL,
    params_json JSONB,                   -- signal parameters used
    cost_scenario TEXT NOT NULL,          -- e.g. 'spot_maker_5bps', 'perps_taker_20bps'
    cv_method TEXT NOT NULL,             -- 'purged_kfold' or 'cpcv'
    n_folds INTEGER NOT NULL,
    embargo_bars INTEGER NOT NULL,
    -- OOS aggregate metrics (mean across folds)
    sharpe_mean DOUBLE PRECISION,
    sharpe_std DOUBLE PRECISION,
    max_drawdown_mean DOUBLE PRECISION,
    max_drawdown_worst DOUBLE PRECISION, -- worst single-fold DD
    total_return_mean DOUBLE PRECISION,
    cagr_mean DOUBLE PRECISION,
    trade_count_total INTEGER,
    turnover DOUBLE PRECISION,
    -- PSR/DSR
    psr DOUBLE PRECISION,                -- PSR on concatenated OOS returns
    dsr DOUBLE PRECISION,                -- DSR across all strategies
    psr_n_obs INTEGER,                   -- number of OOS bars used for PSR
    -- PBO (CPCV only)
    pbo_prob DOUBLE PRECISION,           -- probability of backtest overfitting
    -- Per-fold detail
    fold_metrics_json JSONB,             -- array of per-fold metric dicts
    -- Metadata
    computed_at TIMESTAMPTZ DEFAULT now(),
    UNIQUE(strategy_name, asset_id, tf, params_json, cost_scenario, cv_method)
);
CREATE INDEX idx_bakeoff_strategy ON public.strategy_bakeoff_results (strategy_name, asset_id, tf);
```

Extend `src/ta_lab2/backtests/bakeoff_orchestrator.py` (created in Task 1) with the core walk-forward logic:

1. **run_purged_kfold_backtest(df, signal_fn, params, t1_series, cost, config) -> dict**: Core walk-forward loop:
   - Instantiate `PurgedKFoldSplitter(n_splits=config.n_splits, t1_series=t1_series, embargo_frac=config.embargo_frac)`
   - For each fold: extract train_idx/test_idx, slice df, generate signals on test_df using signal_fn with fixed params, run `run_vbt_on_split`, collect metrics
   - Handle funding_bps_day post-hoc: if cost.funding_bps_day > 0, deduct funding cost from each fold's equity curve before computing metrics. Funding deduction = for each bar where position != 0, deduct `equity * funding_bps_day / 1e4`. Use pf.orders.records to determine position days.
   - Concatenate all OOS returns across folds
   - Compute PSR on concatenated OOS returns via `compute_psr(oos_returns, sr_star=0.0)`
   - Return dict with: sharpe_mean, sharpe_std, max_drawdown_mean, max_drawdown_worst, total_return_mean, cagr_mean, trade_count_total, psr, psr_n_obs, fold_metrics (list of per-fold dicts), oos_returns (the concatenated OOS returns series -- needed for DSR)

2. **run_cpcv_backtest(df, signal_fn, params, t1_series, cost, config) -> dict**: Same as purged_kfold but using CPCVSplitter(n_splits=10, n_test_splits=2, ...) for PBO analysis:
   - Run all 45 combinations
   - For each combination: compute Sharpe on test data
   - PBO = fraction of combinations where Sharpe on the backtest-selected best IS path <= 0 (or worse than random)
   - Return same dict structure as purged_kfold plus pbo_prob

3. **BakeoffOrchestrator class**:
   - `__init__(self, engine, config: BakeoffConfig)`
   - `run(self, strategies: dict[str, dict], asset_ids: list[int], tfs: list[str]) -> pd.DataFrame`
     - strategies maps name -> params dict (e.g. {"ema_trend": {"fast_ema": "ema_21", "slow_ema": "ema_77"}})
     - For each (strategy, asset, tf, cost_scenario, cv_method) combination:
       - Load data, build t1_series
       - Get signal_fn from REGISTRY
       - Ensure required columns via `ensure_for(name, df, params)` from registry
       - Run purged_kfold + cpcv
       - Collect results
     - Compute DSR across all strategies:
       IMPORTANT: `compute_dsr(best_trial_returns, sr_estimates=all_sharpes)`:
       - `best_trial_returns`: The concatenated OOS returns SERIES (pd.Series or np.ndarray) for the strategy with the highest mean OOS Sharpe. This is the actual per-bar returns data, NOT a Sharpe scalar. Get this from the `oos_returns` field of the best strategy's purged_kfold result.
       - `sr_estimates`: A list of mean Sharpe ratio FLOATS (one per strategy). Collect the `sharpe_mean` from each strategy's purged_kfold result. This tells DSR how many trials were run and their distribution, so it can compute the expected maximum SR as the deflation benchmark.
       - Example: if 3 strategies have sharpe_means [1.2, 0.8, 1.05], and ema_trend has the highest (1.2), pass ema_trend's full OOS returns series as `best_trial_returns` and [1.2, 0.8, 1.05] as `sr_estimates`.
     - Write DSR value into each result row (all strategies share the same DSR since it's a cross-strategy metric)
     - Persist all results to strategy_bakeoff_results table

   - `_persist_results(self, results: list[dict])`: Write to strategy_bakeoff_results. Use `json.dumps()` for JSONB columns (params_json, fold_metrics_json). Use `_to_python()` for numpy scalar normalization before SQL binding. INSERT ON CONFLICT DO UPDATE.

4. **Error handling**: Wrap each (strategy, asset, tf, cost) combination in try/except. Log failures but continue to next combination. Report total success/failure at end.

CRITICAL PITFALLS:
- t1_series index MUST be monotonically increasing (cv.py validates this)
- Use pandas comparison operators for tz-aware timestamps (not .values)
- PSR sr_star must be in per-bar units (0.0 for zero benchmark)
- vbt_runner uses `init_cash=1_000.0` -- metrics are normalized
- CostModel.to_vbt_kwargs() does NOT pass funding_bps_day to vectorbt -- must deduct manually post-hoc
- Use `json.dumps(x) if isinstance(x, dict) else x` for JSONB columns before SQL insert
