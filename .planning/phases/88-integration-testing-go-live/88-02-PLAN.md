Create `daily_burn_in_report.py` in the `scripts/integration/` package (created by Plan 01; if Plan 02 executes first, create __init__.py here too with a guard: `if not exists`).

Follow the `weekly_digest.py` pattern:
- NullPool engine via resolve_db_url()
- Structured queries -> formatted output
- Telegram delivery via send_alert/send_message from ta_lab2.notifications.telegram
- ASCII-only output (no UTF-8 box-drawing chars)
- encoding='utf-8' on all file I/O

CLI flags:
- `--burn-in-start DATE` (required): The start date of the burn-in period (YYYY-MM-DD format)
- `--db-url`: Override database URL
- `--no-telegram`: Skip Telegram delivery (stdout only)
- `--dry-run`: Print report but don't send Telegram

Queries to run (each wrapped in try/except, failure = WARN not crash):

1. **Pipeline run status** (pipeline_run_log from Phase 87):
   ```sql
   SELECT status, started_at, completed_at, stages_completed, total_duration_sec
   FROM pipeline_run_log
   WHERE DATE(started_at) = CURRENT_DATE
   ORDER BY started_at DESC
   LIMIT 1
   ```
   NOTE: pipeline_run_log has ONE ROW PER PIPELINE RUN (not per stage). Columns are:
   run_id, started_at, completed_at, status, stages_completed (JSONB), total_duration_sec, error_message.
   There is NO "stage" column and NO "finished_at" column.
   Report: pipeline status today, how many stages completed (from JSONB array length), duration

2. **Order count today** (orders table):
   ```sql
   SELECT COUNT(*) FROM orders
   WHERE created_at >= :burn_in_start AND DATE(created_at) = CURRENT_DATE
   ```

3. **Fill count today** (fills table):
   ```sql
   SELECT COUNT(*) FROM fills
   WHERE filled_at >= :burn_in_start AND DATE(filled_at) = CURRENT_DATE
   ```

4. **Cumulative fills since burn-in start**:
   ```sql
   SELECT COUNT(*) FROM fills WHERE filled_at >= :burn_in_start
   ```

5. **Risk state** (dim_risk_state):
   ```sql
   SELECT trading_state, drift_paused
   FROM dim_risk_state
   WHERE state_id = 1
   ```
   NOTE: PK is state_id (NOT config_id). There is NO kill_switch_active column.
   Kill switch = trading_state = 'halted'. Drift paused = drift_paused = true.
   Report: ACTIVE/HALTED, drift paused yes/no, kill switch = (trading_state = 'halted')

6. **Latest drift metrics** (drift_metrics):
   ```sql
   SELECT tracking_error_5d, tracking_error_30d, metric_date
   FROM drift_metrics
   ORDER BY metric_date DESC
   LIMIT 1
   ```
   NOTE: Columns are tracking_error_5d and tracking_error_30d (NOT tracking_error_pct).
   Date column is metric_date (NOT measured_at). There is no config_id filter needed.
   Report: latest 5d and 30d tracking error %, how recent

7. **Cumulative paper PnL since burn-in start** (fills + positions or a simpler approach):
   ```sql
   SELECT COALESCE(SUM(realized_pnl), 0) as total_pnl
   FROM positions
   WHERE updated_at >= :burn_in_start
   ```
   If positions table doesn't have realized_pnl column, query from fills:
   ```sql
   SELECT COUNT(DISTINCT asset_id) as assets_traded
   FROM orders WHERE created_at >= :burn_in_start
   ```

8. **Signal anomaly gate** (signal_anomaly_log from Phase 87):
   ```sql
   SELECT COUNT(*) as anomaly_count,
          SUM(CASE WHEN blocked THEN 1 ELSE 0 END) as flagged
   FROM signal_anomaly_log
   WHERE checked_at >= :burn_in_start
   ```
   NOTE: The column is "blocked" (BOOLEAN), NOT "is_anomaly".

9. **Burn-in day number**: Calculate from burn_in_start to today (Day N/7)

Report format (ASCII-only, Telegram-compatible HTML):
```
=== DAILY BURN-IN STATUS (Day 3/7) ===
Date: 2026-03-27
Burn-in start: 2026-03-24

PIPELINE
  Status: completed (21 stages, 342s)

TRADING
  Orders today: 14
  Fills today: 12
  Cumulative fills: 45
  Assets traded: 7

RISK
  Trading state: ACTIVE
  Drift paused: No
  Kill switch: No
  Tracking error (5d): 1.23%
  Tracking error (30d): 2.15%

SIGNAL QUALITY
  Anomaly checks: 42, Blocked: 0

BURN-IN VERDICT: ON TRACK
============================================================
```

Verdict logic:
- ON TRACK: trading_state='active', drift_paused=false
- WARNING: any one concern (e.g., high tracking error > 5%)
- STOP: trading_state='halted' OR drift_paused=true

Telegram delivery:
- Use `send_alert(title="Burn-In Day N/7", message=report_text, severity=verdict_severity)`
- severity mapping: ON TRACK -> "info", WARNING -> "warning", STOP -> "critical"
- Respect --no-telegram and --dry-run flags
- Graceful skip if Telegram is not configured (is_configured() returns False)

Error handling:
- Each query section wrapped in try/except
- If a query fails, that section shows "UNAVAILABLE" with the error, script continues
- Only sys.exit(1) if DB connection itself fails
