<objective>
Add a fast-path to EMA computation that avoids loading 15 years of price history when the watermark is recent. For typical daily runs (watermark 1-4 days old), load only the last EMA value per (tf, period) and compute forward using the recursive formula: ema_new = close * alpha + ema_prev * (1 - alpha).

Purpose: Cut EMA refresh from ~59 min (loading full history per ID) to ~10 min (loading only new bars + last EMA).
Output: Fast-path logic in base_ema_refresher + state manager, with fallback to full recompute for stale watermarks.
</objective>

Add a method `load_last_ema_values(self, id_: int, periods: list[int]) -> pd.DataFrame` to EMAStateManager that loads the most recent EMA row per (tf, period) for a given asset ID:

```sql
SELECT DISTINCT ON (tf, period)
    tf, period, ts, ema, ema_bar, roll
FROM {output_table}
WHERE id = :id AND roll = FALSE
ORDER BY tf, period, ts DESC
```

This returns one row per (tf, period) — the last canonical EMA value. The worker can then compute forward from this point.

Also add a method `is_watermark_recent(self, id_: int, threshold_days: int = 7) -> bool` that checks if the minimum watermark across all (tf, period) keys for this ID is within threshold_days of now():
```sql
SELECT MIN(last_canonical_ts) FROM {state_table} WHERE id = :id
```
Returns True if `now() - min_ts < threshold_days`.

Also add `load_recent_bars(self, id_: int, since_ts: timestamp) -> pd.DataFrame` that loads only bars newer than since_ts:
```sql
SELECT ts, close, high, low, open, volume
FROM cmc_price_histories7
WHERE id = :id AND ts >= :since_ts
ORDER BY ts
```
This replaces the full-history load (which loads from watermark-730d).

Modify the EMA computation worker to use the fast-path when watermark is recent:

1. **In `refresh_ema_multi_tf_from_bars.py` `_process_id_worker()`**:
   - After creating engine/state manager, check `mgr.is_watermark_recent(task.id_, threshold_days=7)`
   - If recent: use fast-path
   - If stale or --full-refresh: use existing full-recompute path (no changes)

2. **Fast-path implementation** (in the worker function):
   - Load last EMA values: `mgr.load_last_ema_values(id_, periods)` -> DataFrame with (tf, period, ts, ema, ema_bar)
   - Load recent bars: `mgr.load_recent_bars(id_, since_ts=min_last_ts)` -> DataFrame with daily OHLCV
   - For each (tf, period) combo:
     - Get the last EMA value and its timestamp
     - Get new close prices after that timestamp
     - Compute forward: `ema_new = close * alpha + ema_prev * (1 - alpha)` where `alpha = 2 / (period + 1)`
     - Also compute ema_bar (EMA of bar close, using the bar-boundary close values)
   - Write results using the existing `write_multi_timeframe_ema_to_db()` or the temp-table + COPY + batch upsert pattern already established

3. **In `base_ema_refresher.py`**:
   - Add a `--no-fast-path` CLI flag to force full recompute (useful for debugging)
   - Add a `fast_path_threshold_days` config parameter (default 7)
   - The base class orchestration should pass the fast-path flag through to workers via `WorkerTask.extra_config`

4. **Bar-boundary handling for multi-TF**: The fast-path is simplest for the base (multi_tf) EMA refresh. For daily closes within a bar, the EMA updates on every daily close. The bar-boundary EMA (ema_bar) only updates when a new bar completes (pos_in_bar == tf_days). The fast-path must:
   - Load the last canonical (roll=False) EMA value as the seed
   - Compute new EMAs for each new daily close
   - Mark roll=True for snapshots within partial bars, roll=False for bar completions
   - The existing `write_multi_timeframe_ema_to_db()` already handles this — use it

5. **Fallback safety**: If the fast-path produces fewer rows than expected (e.g., missing bars), log a warning and fall back to full recompute for that ID.

IMPORTANT: Do NOT modify the cal/cal_anchor EMA refreshers in this plan. They use the same base class but have different source data (calendar-aligned bars). The fast-path for those can be added later once the multi_tf fast-path is proven.
