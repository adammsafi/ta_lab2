  <action>
The AMA returns script already has good SQL structure — the _INSERT_SQL uses WINDOW w AS (PARTITION BY id, venue_id, tf, indicator, params_hash ORDER BY ts), so the SQL itself is already batched. The bottleneck is:

1. **Per-(source, id) watermark queries**: Each work unit does a separate `SELECT max(ts) FROM returns_ama_multi_tf_u WHERE id=X AND alignment_source=Y` query. With 5 sources x ~492 IDs = ~2,460 watermark lookups.

2. **Per-(source, id) worker dispatch**: Each (source, id) pair creates its own NullPool engine, runs one SQL, disposes. The engine creation/disposal overhead is significant at 2,460x.

Optimize as follows:

a) **Bulk watermark preload per source**: Before dispatching workers, load ALL watermarks for the source in one query:
   ```sql
   SELECT id, MAX(ts) as last_ts
   FROM returns_ama_multi_tf_u
   WHERE alignment_source = :as_
   GROUP BY id
   ```
   Pass the watermark dict to each worker (avoids per-worker watermark query).

b) **Batch IDs per worker**: Instead of one ID per worker invocation, batch 10-20 IDs per worker call. Each worker creates ONE engine, processes N IDs sequentially, then disposes. This amortizes engine creation.

c) **Skip IDs with no new source data**: Before dispatching, check the source table for max(ts) per ID:
   ```sql
   SELECT id, MAX(ts) as src_max_ts
   FROM ama_multi_tf_u
   WHERE alignment_source = :as_
   GROUP BY id
   ```
   Compare with watermarks. Skip IDs where src_max_ts <= watermark_ts (no new data).

d) **SET work_mem per connection, not per query**: The current code does `SET LOCAL work_mem = '128MB'` per query. Move this to connection-level for batched workers.

e) **Update _worker() signature**: Accept a list of (id, watermark_ts) pairs instead of a single ID. Loop internally. Return a list of result dicts.

f) **Update _process_source()**: Build watermark dict, filter to IDs with new data, chunk into batches, dispatch to pool.

The _INSERT_SQL template itself needs NO changes — it already uses the right PARTITION BY pattern. The optimization is purely in the orchestration layer.

IMPORTANT: Keep the tf_filter parameter working (for --tf single-TF runs). Keep --dry-run working. Keep the 5-source TABLE_MAP iteration. Keep ON CONFLICT DO NOTHING semantics.
  </action>
