  <name>Task 1: Rewrite refresh_returns_ema_multi_tf.py to per-ID batch SQL</name>
  <files>src/ta_lab2/scripts/returns/refresh_returns_ema_multi_tf.py</files>
  <action>
Replace the per-key architecture with per-ID batch processing:

1. **Change key resolution**: Replace `_load_keys()` which returns `List[Tuple[int, str, int, int]]` (id, tf, period, venue_id) with `_load_ids()` returning `List[int]` — just the distinct IDs from the source EMA table.

2. **Bulk watermark preload**: Add `_load_watermarks(engine, state_table, id_) -> dict` that loads ALL watermark rows for a given ID in one query:
   ```sql
   SELECT tf, period, venue_id, last_ts
   FROM {state_table}
   WHERE id = :id
   ```
   Returns dict keyed by (tf, period, venue_id) -> last_ts. Compute `min_last_ts` across all keys for this ID as the seed point.

3. **Source-advance skip**: Before running batch SQL, check if the source table has any rows newer than `min_last_ts` for this ID:
   ```sql
   SELECT 1 FROM {ema_table}
   WHERE id = :id AND alignment_source = :alignment_source
     AND ts > :min_last_ts
   LIMIT 1
   ```
   If no rows found, skip this ID entirely (no new data).

4. **Rewrite `_run_one_key()` -> `_run_one_id()`**: The new function takes just an ID and executes ONE SQL CTE that:
   - Seeds from `min_last_ts - 2 canonical rows` (same seed logic, but the minimum across all keys for this ID)
   - Pulls ALL (tf, period, venue_id) combos from `ema_multi_tf_u` for this ID
   - Uses `PARTITION BY (tf, period, venue_id)` in all LAG() window functions instead of the current bare `ORDER BY ts`
   - The `lagged` CTE becomes:
     ```sql
     LAG(s.ts)  OVER (PARTITION BY s.tf, s.period, s.venue_id ORDER BY s.ts) AS prev_ts_u,
     LAG(s.ema) OVER (PARTITION BY s.tf, s.period, s.venue_id ORDER BY s.ts) AS prev_ema_u,
     -- canonical partition adds s.roll:
     LAG(s.ts)  OVER (PARTITION BY s.tf, s.period, s.venue_id, s.roll ORDER BY s.ts) AS prev_ts_c,
     ```
   - The `to_insert` CTE filters: for each (tf, period, venue_id), only rows with `ts > last_ts` from the preloaded watermarks. Use a LEFT JOIN to the watermarks or pass them as a VALUES clause.
   - INSERT uses ON CONFLICT DO UPDATE SET (same upsert pattern)

5. **State update**: After the batch INSERT, update the state table in bulk:
   ```sql
   INSERT INTO {state_table} (id, tf, period, venue_id, last_ts, updated_at)
   SELECT id, tf, period, venue_id, MAX(ts), now()
   FROM (just-inserted rows or from out_table)
   GROUP BY id, tf, period, venue_id
   ON CONFLICT (id, tf, period, venue_id) DO UPDATE SET
     last_ts = GREATEST(EXCLUDED.last_ts, {state_table}.last_ts),
     updated_at = now()
   ```

6. **Multiprocessing wrapper**: Update `_run_one_key_mp` -> `_run_one_id_mp` taking just `(db_url, tables, id_)`.

7. **main() changes**:
   - Load distinct IDs instead of all keys
   - Print ID count (should be ~492 for --ids all)
   - Loop over IDs, not keys
   - Keep --full-refresh support (DELETE scope changes to per-ID across all tf/period/venue)
   - Keep --workers support

8. **Watermark passing strategy**: The simplest approach is to pass watermarks into the SQL via a CTE built from VALUES:
   ```sql
   wm AS (
     VALUES (tf1, period1, venue1, last_ts1),
            (tf2, period2, venue2, last_ts2), ...
   )
   ```
   Then JOIN in the `to_insert` CTE to filter by per-key watermark. If no watermark exists for a key, include it with NULL last_ts (meaning: include all rows).

CRITICAL: The existing SQL semantics must be preserved exactly:
- Unified LAG (prev_*_u) = LAG across ALL roll values within same (tf, period, venue_id)
- Canonical LAG (prev_*_c) = LAG within same roll partition AND same (tf, period, venue_id)
- The delta2/delta_ret second-pass LAG must also include PARTITION BY (tf, period, venue_id)
- alignment_source filter on source reads (prevents cross-source contamination)

Keep the --full-refresh path working: when active, DELETE all rows for this ID, reset all state rows, then recompute.
  </action>
