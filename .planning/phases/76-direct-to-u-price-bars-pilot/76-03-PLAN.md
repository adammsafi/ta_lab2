Run a verification query that compares row counts in the 5 siloed tables against the _u table filtered by alignment_source. This confirms no data loss or duplication.

Execute this SQL against the database (using TARGET_DB_URL):

```sql
WITH siloed AS (
    SELECT 'multi_tf' AS alignment_source,
           COUNT(*) AS siloed_count
    FROM public.price_bars_multi_tf
    UNION ALL
    SELECT 'multi_tf_cal_us',
           COUNT(*)
    FROM public.price_bars_multi_tf_cal_us
    UNION ALL
    SELECT 'multi_tf_cal_iso',
           COUNT(*)
    FROM public.price_bars_multi_tf_cal_iso
    UNION ALL
    SELECT 'multi_tf_cal_anchor_us',
           COUNT(*)
    FROM public.price_bars_multi_tf_cal_anchor_us
    UNION ALL
    SELECT 'multi_tf_cal_anchor_iso',
           COUNT(*)
    FROM public.price_bars_multi_tf_cal_anchor_iso
),
unified AS (
    SELECT alignment_source,
           COUNT(*) AS u_count
    FROM public.price_bars_multi_tf_u
    GROUP BY alignment_source
)
SELECT
    s.alignment_source,
    s.siloed_count,
    COALESCE(u.u_count, 0) AS u_count,
    CASE
        WHEN s.siloed_count = COALESCE(u.u_count, 0) THEN 'MATCH'
        WHEN COALESCE(u.u_count, 0) > s.siloed_count THEN 'U_HAS_MORE'
        ELSE 'MISMATCH'
    END AS status
FROM siloed s
LEFT JOIN unified u ON u.alignment_source = s.alignment_source
ORDER BY s.alignment_source;
```

Expected result: All 5 rows show 'MATCH' or 'U_HAS_MORE' (U_HAS_MORE is acceptable if builders have run since the initial sync and added new data directly to _u).

If any row shows 'MISMATCH' (u_count < siloed_count), investigate:
- Could be timing (siloed table got new data after last sync but before builder redirect)
- In that case, the next builder run will bring _u up to date
- Log the discrepancy but do not fail -- the builders are now writing directly to _u

Print a summary table with all 5 alignment_source counts and status.
