**drift_monitor.py:**

```python
class DriftMonitor:
    """Daily drift comparison between paper executor and backtest replay."""
```

1. `__init__(self, engine: Engine)`: Store engine. Import CostModel from ta_lab2.backtests.costs and SignalBacktester from ta_lab2.scripts.backtests.backtest_from_signals at init time (or defer to method-level imports to avoid circular imports -- check what works).

2. `run(self, paper_start_date: str, dry_run: bool = False) -> list[DriftMetrics]`:
   - today = datetime.now(timezone.utc).date().isoformat()
   - configs = self._load_active_executor_configs() -- query dim_executor_config WHERE is_active=TRUE. Return list of dicts with config_id, signal_id, signal_type, asset_id (from dim_executor_config). NOTE: dim_executor_config may store asset_ids differently; check the actual schema. If the config has a single asset_id, use it. If it processes multiple assets (from the signal table), load from dim_signals.
   - For each config: try/except to run _check_strategy_drift(config, paper_start_date, today). On success, write metrics to DB (unless dry_run). Check threshold. Append to results.
   - After all configs: call _refresh_summary_view() (unless dry_run).
   - Check escalation: check_drift_escalation(engine).
   - Return results list.

3. `_load_active_executor_configs(self) -> list[dict]`:
   - Query: SELECT config_id, signal_id, signal_type FROM dim_executor_config WHERE is_active = TRUE
   - For each config, determine asset_ids by querying the signal table: SELECT DISTINCT id FROM cmc_signals_{signal_type} WHERE signal_id = :signal_id. This gives the list of assets for that strategy.
   - Use SIGNAL_TABLE_MAP from ta_lab2.executor.signal_reader to map signal_type -> table name. Guard against SQL injection by validating table name against the map.
   - Return list of dicts: [{config_id, signal_id, signal_type, asset_ids: [int]}]

4. `_check_strategy_drift(self, config: dict, asset_id: int, paper_start: str, today: str) -> DriftMetrics`:
   - Build CostModel matching the executor config (query dim_executor_config for fee_bps, slippage settings).
   - Run PIT replay: Use SignalBacktester.run_backtest(signal_type, signal_id, asset_id, start_ts, end_ts). The backtester already handles signal loading and cost modeling.
   - Run current-data replay: Same call but with current data (no PIT filter needed -- the backtester uses current data by default).
   - Load paper fills: SELECT from cmc_fills JOIN cmc_orders WHERE signal_id and asset_id and date range. Similar to ParityChecker._load_executor_fills.
   - Compute daily P&L arrays from paper fills and replay results.
   - Call compute_drift_metrics() from drift_metrics.py.
   - Return DriftMetrics.

   NOTE on PIT replay: For V1, the "PIT replay" and "current data replay" will likely produce identical results because data revisions are rare in crypto. The framework captures the delta as data_revision_pnl_diff. If PIT snapshots are not yet populated (executor hasn't been patched to collect them), run both replays with current data and set data_revision_pnl_diff=0. Log a WARNING that PIT snapshots are not yet available.

5. `_aggregate_daily_pnl(self, fills: list[dict], start_date: str, end_date: str) -> np.ndarray`:
   - Convert fills to daily P&L by date. For each day in range, sum fill P&L for that day (or 0 if no fills).
   - Returns numpy array of daily P&L values.

6. `_write_metrics(self, metrics: DriftMetrics) -> None`:
   - INSERT INTO cmc_drift_metrics with ON CONFLICT (metric_date, config_id, asset_id) DO UPDATE SET ... for all non-PK columns.
   - Use sqlalchemy text() with named parameters. json.dumps for any JSONB fields.

7. `_refresh_summary_view(self) -> None`:
   - First check if v_drift_summary has any rows. If empty, use non-concurrent refresh: REFRESH MATERIALIZED VIEW public.v_drift_summary
   - If has rows, use: REFRESH MATERIALIZED VIEW CONCURRENTLY public.v_drift_summary
   - Wrap in try/except -- view refresh failure should not crash the monitor.
