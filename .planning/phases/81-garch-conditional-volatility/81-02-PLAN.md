<task type="auto">
  <name>Task 2: Daily GARCH refresh script</name>
  <files>src/ta_lab2/scripts/garch/refresh_garch_forecasts.py</files>
  <action>
Create the main daily refresh CLI script. This is invoked via `subprocess.run` from `run_daily_refresh.py` (wired in Plan 05).

**CLI interface (argparse):**
- `--ids`: Comma-separated asset IDs or "all" (default: "all")
- `--tf`: Timeframe (default: "1D")
- `--venue-id`: Venue ID (default: 1 = CMC_AGG)
- `--db-url`: Database URL override (default: from config/env)
- `--min-obs`: Minimum observations (default: 126)
- `--verbose`: Verbose output
- `--dry-run`: Show what would execute without running

**Main logic flow:**
1. Parse args, resolve DB URL using `ta_lab2.scripts.refresh_utils.resolve_db_url`
2. Create engine (use `NullPool` -- project convention for scripts)
3. Create GARCHStateManager and ensure state table
4. Load list of asset IDs: if "all", query `SELECT DISTINCT id FROM returns_bars_multi_tf WHERE venue_id = {venue_id} AND tf = '{tf}'`
5. For each asset_id (sequential, no multiprocessing -- research pitfall #7):
   a. Load returns: `SELECT ts, ret_log FROM returns_bars_multi_tf WHERE id = :id AND venue_id = :venue_id AND tf = :tf ORDER BY ts ASC`
   b. Skip if len(returns) < min_obs (log warning)
   c. Call `fit_all_variants(returns_array, min_obs=min_obs)` from garch_engine
   d. For each model_type result:
      - Insert diagnostics row into garch_diagnostics (capturing run_id via RETURNING)
      - If converged: insert forecast rows (h=1 and h=5) into garch_forecasts with model_run_id
      - If NOT converged and carry-forward applicable: query last valid forecast from garch_forecasts, apply decay logic (up to 5 days), insert with forecast_source='carry_forward'
      - If NOT converged and beyond decay window: use GK vol from features table if available, insert with forecast_source='fallback_gk'
      - Update state via GARCHStateManager
6. After all assets: `REFRESH MATERIALIZED VIEW CONCURRENTLY garch_forecasts_latest`
7. Print summary: N assets processed, N converged (by model type), N fell back, N skipped

**DB writes (batch upsert pattern):**
Use the project-standard temp table + ON CONFLICT pattern for both garch_forecasts and garch_diagnostics:
- For garch_forecasts: batch all forecast rows for the current run into a temp table, then `INSERT INTO garch_forecasts SELECT * FROM temp_garch_fc ON CONFLICT (id, venue_id, ts, tf, model_type, horizon) DO UPDATE SET cond_vol = EXCLUDED.cond_vol, forecast_source = EXCLUDED.forecast_source, model_run_id = EXCLUDED.model_run_id, created_at = EXCLUDED.created_at`
- For garch_diagnostics: use single INSERT with RETURNING run_id (each row is unique by run_id BIGSERIAL)

**Critical anti-patterns to avoid:**
- Do NOT fit on unscaled returns (garch_engine handles this, but verify)
- Do NOT use `_u` union tables for returns (use `returns_bars_multi_tf` filtered)
- Do NOT use multiprocessing within this script (NullPool + Windows issue)
- Store vol (not variance) in cond_vol column
- Log but do NOT raise on per-asset failures; continue to next asset
  </action>
