<objective>
Build a cross-asset CTF composite script that computes three composite types (sentiment, relative-value, leader-follower) and persists them to the `ctf_composites` table. Materialize top composites into the `features` table for downstream consumption.

Purpose: Individual CTF features per asset are noisy. Cross-asset aggregation creates market-wide signals: sentiment (bullish/bearish consensus), relative-value (which assets are rich/cheap), and leader-follower (which assets lead market moves). These are the composite signals that feed BL and portfolio optimization.

Output: One CLI script, one config file, composites in DB + materialized in features.
</objective>

  <files>src/ta_lab2/scripts/features/refresh_ctf_composites.py</files>
  <action>
Create `refresh_ctf_composites.py` as a CLI script that computes cross-asset CTF composites.

**Script structure:**

1. **Load config** from `configs/ctf_composites_config.yaml`.

2. **Load promoted features** from `configs/feature_selection.yaml` (the `ctf_promoted.features` list from 98-01). Fall back to querying `ic_results` if YAML section missing.

3. **For each base_tf in config.base_tfs:**
   a. Load CTF data for ALL tier-1 assets using `load_ctf_features()` for each asset
   b. Build a multi-asset pivot DataFrame: rows=timestamps, columns=`{asset_id}_{feature_name}` or use MultiIndex

4. **Compute sentiment_mean composite:**
   - For each promoted CTF feature, compute cross-asset mean at each timestamp
   - Result: DataFrame with columns like `sentiment_mean_{feature_name}`, indexed by ts
   - Each row = one `(ts, tf, venue_id, composite_name=f"sentiment_mean_{feat}", method="cross_asset_mean")` entry

5. **Compute sentiment_pca composite:**
   - For each promoted CTF feature: pivot across assets, standardize (StandardScaler), fit PCA(n_components=1)
   - Sign-correct: `loadings = pca.components_[0]; dominant_sign = np.sign(loadings[np.abs(loadings).argmax()]); scores *= dominant_sign`
   - Result: one column per feature, `composite_name=f"sentiment_pca_{feat}"`, method="pca_1"
   - Skip if fewer than `min_assets` have non-NaN values at a given timestamp; drop NaN rows before PCA

6. **Compute relative_value composite:**
   - For each (ts, feature): compute cross-sectional z-score across assets
   - `rv = (asset_value - cross_section_mean) / cross_section_std`
   - This is PER-ASSET (unlike sentiment which is market-wide), so store with `composite_name=f"rv_{feat}_{asset_id}"`
   - Actually, relative-value is a per-asset signal, not a single market aggregate. Store as `composite_name=f"relative_value_{feat}"` with one row per (ts, asset). Use the `value` column for the z-score and add asset context in composite_name or use a different schema approach.
   - DECISION: Store relative-value per-asset in features table directly (it's an asset-level signal). In ctf_composites, store the cross-section stats (mean, std) as method="cs_stats". The per-asset z-score goes directly into the features table UPDATE.

7. **Compute leader_follower composite:**
   - For each pair of promoted features: compute `lead_lag_max_corr()` from `regimes.comovement` across asset pairs
   - Identify top N leaders (assets that consistently lead others) based on average best_corr at negative lags
   - Run `statsmodels.tsa.stattools.grangercausalitytests` on top leaders as validation (lag order 1-5, select by min AIC)
   - Store: `composite_name="leader_score"`, `method="lagged_corr"`, `value` = leader score for each asset

8. **Persist to ctf_composites**: Use temp table + INSERT ON CONFLICT upsert pattern from cross_asset.py:
   ```python
   df.to_sql("_tmp_ctf_composites", engine, schema="public", if_exists="replace", index=False)
   conn.execute(text("""
       INSERT INTO public.ctf_composites (ts, tf, venue_id, composite_name, method, value, n_assets)
       SELECT ts, tf, venue_id, composite_name, method, value, n_assets
       FROM public._tmp_ctf_composites
       ON CONFLICT (ts, tf, venue_id, composite_name, method) DO UPDATE SET
           value = EXCLUDED.value, n_assets = EXCLUDED.n_assets, computed_at = now()
   """))
   conn.execute(text("DROP TABLE IF EXISTS public._tmp_ctf_composites"))
   ```

9. **Materialize to features**: For composites in `config.materialize_to_features`, UPDATE the features table with sentiment_mean and relative_value columns. Use the same UPDATE pattern as refresh_ctf_promoted.py. Only materialize the market-wide composites (sentiment_mean) and per-asset z-scores (relative_value) — leader_follower and PCA are research outputs.

   NOTE: The columns being materialized into features must exist (added in the 98-01 migration). Add composite column names (e.g., `ctf_sentiment_mean`, `ctf_relative_value_z`) to the migration's dynamic column list, or add them as fixed columns in the migration alongside the CTF promoted columns. Prefer adding a small set of fixed composite columns: `ctf_sentiment_mean`, `ctf_relative_value_z`.

**CLI interface:**
```python
if __name__ == "__main__":
    parser = argparse.ArgumentParser(description="Compute and persist CTF cross-asset composites")
    parser.add_argument("--dry-run", action="store_true")
    parser.add_argument("--composite", type=str, choices=["sentiment_mean", "sentiment_pca", "relative_value", "leader_follower"], help="Run single composite")
    parser.add_argument("--base-tf", type=str, default="1D")
    args = parser.parse_args()
```

**Key constraints:**
- Use `NullPool` if multiprocessing (but likely not needed — sequential over assets is fine for ~7 assets)
- Handle missing data: skip timestamps where fewer than `min_assets` have values
- PCA sign correction is mandatory (research column, but must be consistent)
