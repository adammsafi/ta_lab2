Create a single Alembic migration file chaining from `q1r2s3t4u5v6` with revision `r2s3t4u5v6w7`.

The migration must do four things:

1. **Add CTF promoted columns to `features` table.** Before adding columns, query `ic_results` to find CTF features passing IC > 0.02 (cross-asset median). Use this SQL pattern to discover the feature list dynamically:
   ```sql
   SELECT feature FROM public.ic_results
   WHERE horizon = 1 AND return_type = 'arith' AND regime_col = 'all' AND regime_label = 'all'
     AND ic IS NOT NULL
     AND (feature LIKE '%_slope' OR feature LIKE '%_divergence' OR feature LIKE '%_agreement'
          OR feature LIKE '%_crossover' OR feature LIKE '%_ref_value' OR feature LIKE '%_base_value')
   GROUP BY feature
   HAVING PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY ABS(ic)) > 0.02
   ```
   For each feature name returned, add `op.add_column("features", sa.Column(feature_name, sa.Float(), nullable=True), schema="public")`. If the query returns 0 features (unlikely but possible), log a warning and continue with empty column list.

   IMPORTANT: Feature names like `rsi_14_7d_slope` must be valid PostgreSQL column names. CTF feature names use only lowercase alphanumeric + underscores, so they are safe.

2. **Create `dim_feature_selection_asset` table** (separate from dim_feature_selection to avoid TRUNCATE hazard):
   ```python
   op.create_table(
       "dim_feature_selection_asset",
       sa.Column("feature_name", sa.Text(), nullable=False),
       sa.Column("asset_id", sa.Integer(), nullable=False),
       sa.Column("tier", sa.Text(), nullable=False, server_default="asset_specific"),
       sa.Column("ic_ir_mean", sa.Numeric(), nullable=True),
       sa.Column("pass_rate", sa.Numeric(), nullable=True),
       sa.Column("stationarity", sa.Text(), nullable=True),
       sa.Column("selected_at", sa.DateTime(timezone=True), server_default=sa.func.now()),
       sa.Column("yaml_version", sa.Text(), nullable=True),
       sa.Column("rationale", sa.Text(), nullable=True),
       sa.PrimaryKeyConstraint("feature_name", "asset_id"),
       schema="public",
   )
   ```

3. **Create `ctf_composites` table:**
   ```python
   op.create_table(
       "ctf_composites",
       sa.Column("ts", sa.DateTime(timezone=True), nullable=False),
       sa.Column("tf", sa.Text(), nullable=False),
       sa.Column("venue_id", sa.SmallInteger(), nullable=False, server_default="1"),
       sa.Column("composite_name", sa.Text(), nullable=False),
       sa.Column("method", sa.Text(), nullable=False),
       sa.Column("value", sa.Float(), nullable=True),
       sa.Column("n_assets", sa.Integer(), nullable=True),
       sa.Column("computed_at", sa.DateTime(timezone=True), server_default=sa.func.now()),
       sa.PrimaryKeyConstraint("ts", "tf", "venue_id", "composite_name", "method"),
       schema="public",
   )
   ```

4. **Create `lead_lag_ic` table:**
   ```python
   op.create_table(
       "lead_lag_ic",
       sa.Column("asset_a_id", sa.Integer(), nullable=False),
       sa.Column("asset_b_id", sa.Integer(), nullable=False),
       sa.Column("feature", sa.Text(), nullable=False),
       sa.Column("horizon", sa.Integer(), nullable=False),
       sa.Column("tf", sa.Text(), nullable=False),
       sa.Column("venue_id", sa.SmallInteger(), nullable=False, server_default="1"),
       sa.Column("ic", sa.Numeric(), nullable=True),
       sa.Column("ic_p_value", sa.Numeric(), nullable=True),
       sa.Column("ic_p_bh", sa.Numeric(), nullable=True),
       sa.Column("is_significant", sa.Boolean(), nullable=True),
       sa.Column("n_obs", sa.Integer(), nullable=True),
       sa.Column("computed_at", sa.DateTime(timezone=True), server_default=sa.func.now()),
       sa.PrimaryKeyConstraint("asset_a_id", "asset_b_id", "feature", "horizon", "tf", "venue_id"),
       schema="public",
   )
   ```

The `downgrade()` must reverse all four operations in reverse order: drop lead_lag_ic, drop ctf_composites, drop dim_feature_selection_asset, remove CTF columns from features.

For the dynamic column list in downgrade, hardcode the column names discovered during upgrade (store them in a module-level constant `_CTF_PROMOTED_COLS` populated by the upgrade query, or hardcode them after running the query once during development).

IMPORTANT: The dynamic IC query happens in `upgrade()` using `op.get_bind().execute()`. This is a data-dependent migration. Document this prominently in the docstring.

Create `refresh_ctf_promoted.py` as a standalone CLI script following the existing pattern from `microstructure_feature.py`.

**Script structure:**

1. **Pre-flight column check** (CRITICAL): At script start, query `information_schema.columns` for the features table and compare against the expected CTF promoted column list. If any columns are missing, raise `RuntimeError("Run Alembic migration first. Missing columns: {missing}")`. This prevents the silent-drop pitfall.

2. **Discover promoted features**: Query `ic_results` using the same PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY ABS(ic)) > 0.02 query from the migration. Return list of feature names + their median_abs_ic and n_assets for YAML registration.

3. **For each tier-1 asset (all distinct `id` in features table where venue_id=1)**:
   a. Load CTF data using `load_ctf_features(conn, asset_id=id, base_tf=tf, ...)` for each base_tf
   b. Pivot the CTF feature columns to match the promoted feature names
   c. Build a DataFrame with columns: `id, ts, tf, venue_id, alignment_source` + promoted CTF feature columns
   d. UPDATE features rows using the microstructure_feature.py pattern:
      ```python
      set_clauses = ", ".join(f"{col} = :{col}" for col in ctf_promoted_cols)
      sql = text(f"""
          UPDATE public.features
          SET {set_clauses}
          WHERE id = :id AND ts = :ts AND tf = :tf
            AND venue_id = :venue_id AND alignment_source = :alignment_source
      """)
      ```
   e. Batch execute in chunks of 5000 rows

4. **Update feature_selection.yaml**: After all UPDATEs complete, read the existing YAML, add/overwrite a `ctf_promoted:` top-level section:
   ```yaml
   ctf_promoted:
     generated_at: "2026-03-31T..."
     ic_threshold: 0.02
     ic_aggregation: cross_asset_median
     source: ic_results
     features:
     - name: rsi_14_7d_slope
       median_abs_ic: 0.0312
       n_assets: 6
       base_tf: 1D
       source_ctf_config: ctf_config.yaml
     - ...
   ```

**CLI interface:**
```python
if __name__ == "__main__":
    import argparse
    parser = argparse.ArgumentParser(description="Promote CTF features to features table")
    parser.add_argument("--dry-run", action="store_true", help="Show features that would be promoted without writing")
    parser.add_argument("--ic-threshold", type=float, default=0.02, help="IC threshold for promotion")
    parser.add_argument("--base-tf", type=str, default=None, help="Specific base_tf to process (default: all)")
    args = parser.parse_args()
```

**Key constraints:**
- Use `alignment_source='multi_tf'` when loading CTF features (matches features table convention)
- Use `venue_id=1` (CMC_AGG default)
- DO NOT use `save_to_db()` from `feature_selection.py` (TRUNCATE hazard)
- DO NOT use DELETE+INSERT (breaks other columns). Use UPDATE only
- Handle NaN values: CTF features may be NaN for some timestamps. UPDATE should set NULL for NaN values (use `numpy.where(pd.isna(v), None, v)` when building row dicts)
- Log progress: "Promoted {N} CTF features for asset {id} tf={tf}: {X} rows updated"

**Dry-run mode:** When `--dry-run` is set, query the IC results and print the list of features that would be promoted with their median IC, then exit without writing anything.
