Create `refresh_cmc_cross_asset_corr.py` following the same structural patterns as refresh_cmc_asset_stats.py.

**Constants:**
- WINDOWS = [30, 60, 90, 252]  # User-approved scope (see 41-CONTEXT.md: "Same windows as stats")
- TABLE_NAME = "cmc_cross_asset_corr"
- STATE_TABLE = "cmc_cross_asset_corr_state"
- SOURCE_TABLE = "cmc_returns_bars_multi_tf"

**Core computation function: `compute_pairwise_rolling_corr(ret_a, ret_b, window, ts_index)`**
- Input: Two pd.Series (ret_arith for asset A and B), integer window, DatetimeIndex
- For each position i from 0 to len(ts_index)-1:
  - If i < window - 1: append all NaN row
  - Extract slices: a_slice = ret_a.iloc[i - window + 1 : i + 1], same for b
  - Compute intersection mask: both non-null
  - Count n = intersection count. If n < window: append NaN row with n_obs=n
  - Else: call scipy.stats.pearsonr(a_vals, b_vals) and scipy.stats.spearmanr(a_vals, b_vals)
  - Append (pearson_r, pearson_p, spearman_r, spearman_p, n_obs) as a row
- Return DataFrame with columns: pearson_r, pearson_p, spearman_r, spearman_p, n_obs

CRITICAL: Use `spearmanr(a_vals, b_vals).statistic` and `.pvalue` (not positional indexing) -- scipy SpearmanrResult is a named tuple with these attributes. Same for pearsonr.

**Pair generation:**
- Load all asset IDs from dim_assets (or filter by --ids)
- Generate canonical pairs: `[(a, b) for a in ids for b in ids if a < b]` -- ensures id_a < id_b
- For each TF, process all pairs

**Data loading strategy per TF:**
- Load ALL assets' ret_arith for this TF in one query (efficient single scan):
  ```sql
  SELECT id, ts, ret_arith FROM cmc_returns_bars_multi_tf
  WHERE tf = :tf AND roll = FALSE AND id IN :ids
  ORDER BY ts
  ```
- Pivot to wide DataFrame with ts as index, one column per asset id
- For each pair (id_a, id_b), pass columns id_a and id_b to compute_pairwise_rolling_corr

**Watermark-based incremental refresh:**
- Read last_timestamp from cmc_cross_asset_corr_state per (id_a, id_b, tf)
- For each pair+tf: determine start_ts = min(last_timestamp) - lookback (max window = 252)
- When loading data for a TF, use the global minimum watermark across all pairs for that TF
- After computation, DELETE + INSERT for affected rows
- UPDATE state table

**Worker function for multiprocessing (TF-level parallelism):**
- `_worker_tf(tf, ids, windows, db_url, full_rebuild)` processes all pairs for one TF
- Creates NullPool engine per worker
- Returns summary (tf, n_pairs, n_rows, success, error_msg)

**Materialized view refresh (at end of main):**
- After all TFs complete, run:
  ```sql
  REFRESH MATERIALIZED VIEW CONCURRENTLY public.cmc_corr_latest;
  ```
- This uses CONCURRENTLY (requires unique index, which we created in Plan 01)
- Wrap in try/except -- if view is empty (first run with no data), CONCURRENTLY may fail; fall back to non-concurrent refresh

**CLI (argparse):**
- `--ids` (comma-separated or "all", default "all")
- `--tf` (single TF filter, optional)
- `--windows` (comma-separated, default "30,60,90,252")
- `--full-rebuild` (ignore watermarks)
- `--dry-run`
- `--continue-on-error`
- `--workers` (default 4)
- `--db-url`

**Main flow:**
1. Parse args, resolve DB URL
2. Load asset IDs and TF list
3. Generate canonical pair list
4. Run TF-level workers via multiprocessing.Pool
5. Refresh materialized view
6. Print summary

Use NullPool everywhere. Import from refresh_utils and dim_timeframe.
Module-level `if __name__ == "__main__"`.
